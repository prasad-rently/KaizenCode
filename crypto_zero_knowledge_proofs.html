🎭 Zero-Knowledge Proofs — Prove Without Revealing

Convince someone that a statement is true while telling them nothing else — not the secret, not even a hint. Prove you know a password without sending it; prove you're over 18 without showing your birthday; prove a transaction is valid without revealing the amounts.

A bonus module in the Cryptography topic, and the most mind-bending idea here. It builds on the same discrete-log hardness as ECDH and signatures — in fact the protocol below is the direct ancestor of Schnorr signatures.

1The idea — the cave

Analogy — Ali Baba's cave. A ring-shaped cave has a magic door at the back that only opens with a secret word. Peggy claims she knows the word; Victor wants proof, but Peggy won't say it.

  • Victor waits outside. Peggy walks in and takes the left or right passage (Victor can't see which).
  • Victor shouts which side he wants her to come out of.
  • If Peggy really knows the word, she can always comply — opening the door if needed. If she was bluffing, she only had a 50% chance of already being on the right side.

Do it once and a cheat might get lucky. Do it 20 times and a bluffer's odds of passing every round are 1 in 2²⁰ — about one in a million. Victor ends up certain Peggy knows the word, yet he never learned it.

2The three properties

  • Completeness — if the statement is true and both play honestly, the verifier is convinced.
  • Soundness — if it's false, a cheating prover is caught (except with negligible probability).
  • Zero-knowledge — the verifier learns nothing beyond "the statement is true." The whole transcript could have been faked without the secret, so it can't contain any.

3A real protocol — Schnorr

The cave in real math: public values are a prime p, a base g, and y = gˣ mod p. Peggy proves she knows the secret exponent x (the "password") without revealing it, in a three-move commit → challenge → response dance:

   PROVER (knows x)                         VERIFIER
   pick random r
   t = gʳ mod p          ───(commit t)───▶
                         ◀──(challenge c)──   pick random c
   s = r + c·x           ────(response s)─▶
                                              check:  gˢ  ==  t · yᶜ  (mod p)

It works because gˢ = g^(r + c·x) = gʳ · (gˣ)ᶜ = t · yᶜ. The commitment t must be sent before the challenge — that's what stops a cheat from working backwards. And r blinds x completely, so s leaks nothing.

4See it: honest prover vs cheater

JavaScript · runnable
// Schnorr ZKP: prove knowledge of x where y = gˣ mod p — without revealing x
const p = 2147483647n, g = 7n;        // public parameters
const mod = (a, m) => ((a % m) + m) % m;
const modpow = (b, e, m) => { let r = 1n; b = mod(b, m); while (e > 0n) { if (e & 1n) r = r * b % m; b = b * b % m; e >>= 1n; } return r; };
const rand = lim => { const u = new Uint32Array(2); crypto.getRandomValues(u); return mod((BigInt(u[0]) << 32n) | BigInt(u[1]), lim); };

const x = 123456789n;               // the prover's SECRET
const y = modpow(g, x, p);           // public: y = gˣ mod p
console.log('public y = gˣ mod p =', y.toString());

const round = (secretUsed) => {
  const r = rand(p - 1n) || 1n;
  const t = modpow(g, r, p);          // 1) commit  (before the challenge!)
  const c = rand(p - 1n);            // 2) verifier's random challenge
  const s = mod(r + c * secretUsed, p - 1n);  // 3) response
  return modpow(g, s, p) === mod(t * modpow(y, c, p), p);  // verify gˢ == t·yᶜ
};

console.log('honest prover (uses real x) → verified:', round(x));
console.log('cheater (uses wrong secret) → verified:', round(x + 1n));
console.log('the verifier only ever saw t, c, s — never x');

Try it yourself — prove you know a secret

Pick a secret x. The page publishes y = gˣ and then runs the full protocol. Prove honestly and it verifies; try to fake it (proving knowledge of the same y without the real x) and it's rejected. Notice the transcript only ever contains t, c, s — never your secret.

5Making it non-interactive (Fiat–Shamir)

The protocol above needs a live back-and-forth. The Fiat–Shamir transform removes that: instead of waiting for the verifier's random challenge, the prover computes the challenge as a hash of the commitment (and message): c = H(g, y, t). Now the whole proof is a single self-contained message anyone can check later — no interaction. Apply this exact trick to Schnorr and you get Schnorr signatures. A digital signature is a non-interactive zero-knowledge proof that you know the private key.

6Where zero-knowledge shows up

  • Privacy coins — Zcash uses zk-SNARKs to prove a transaction is valid (inputs = outputs, no double-spend) while hiding sender, receiver, and amount.
  • Blockchain scaling — zk-rollups post one succinct proof that thousands of transactions were executed correctly, instead of re-running them all on-chain.
  • Identity & credentials — prove "I'm over 18" or "I'm a citizen" from a signed credential without revealing the underlying data.
  • Authentication — prove knowledge of a password/key to a server that never sees it.
Nuances worth knowing. The interactive Schnorr protocol is honest-verifier zero-knowledge; real deployments harden it. As with ECDSA, the random nonce r must never repeat or be predictable — reuse leaks the secret. And "zero-knowledge" is about not revealing the witness; you still must be able to express your statement as a checkable relation. Modern zk-SNARKs/STARKs generalize this to prove arbitrary computations, at the cost of considerable machinery (and sometimes a trusted setup).
Try it: (1) Prove honestly a few times — each run shows a different commitment t and response s, yet always verifies, and never repeats info about x. (2) Try to fake it — rejected every time. (3) Change x and prove again: the public y changes, but the verifier still learns only that you know some valid x.
