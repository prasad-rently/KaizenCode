Databases

Where your data lives permanently — queryable, relational, and built for scale.

1SQL vs NoSQL — the fundamental choice

Before writing a query, choose your database type. This architectural decision is hard to reverse once data is in production.

SQL (Relational)
Tables with fixed columns. Rows relate via foreign keys. ACID transactions. Strong consistency.

PostgreSQL · MySQL · SQLite
NoSQL (Document)
JSON documents. No fixed schema. Flexible, fast writes. Weak cross-document transactions.

MongoDB · Firestore · DynamoDB
Mobile analogy: SQL = Core Data with a strict managed object model. NoSQL = storing raw JSON in a folder — flexible but you enforce the shape yourself.
Use SQL whenUse NoSQL when
Complex relationships / JOINs neededFlexible or evolving schema
Financial / transactional dataHigh-volume writes / logs / events
Data integrity is criticalPrototyping fast or early-stage product

2PostgreSQL — tables, rows, columns

-- Create a table
CREATE TABLE users (
  id         SERIAL PRIMARY KEY,
  name       VARCHAR(100) NOT NULL,
  email      VARCHAR(255) UNIQUE NOT NULL,
  role       VARCHAR(20)  DEFAULT 'user',
  created_at TIMESTAMPTZ   DEFAULT NOW()
);

The 4 core SQL operations

-- INSERT
INSERT INTO users (name, email) VALUES ('Gokul', 'g@example.com');

-- SELECT
SELECT * FROM users;
SELECT name, email FROM users WHERE id = 1;
SELECT * FROM users ORDER BY created_at DESC LIMIT 10 OFFSET 0;

-- UPDATE
UPDATE users SET name = 'Gokul Prasad' WHERE id = 1;

-- DELETE
DELETE FROM users WHERE id = 1;
Never run UPDATE or DELETE without a WHERE clause — it affects every row in the table.

JOINs — relating tables

-- posts table with a foreign key to users
CREATE TABLE posts (
  id      SERIAL PRIMARY KEY,
  user_id INT REFERENCES users(id) ON DELETE CASCADE,
  title   VARCHAR(255),
  body    TEXT
);

-- INNER JOIN: get posts with their author's name
SELECT posts.title, users.name AS author
FROM posts
JOIN users ON posts.user_id = users.id;

-- LEFT JOIN: all users, even those with no posts
SELECT users.name, COUNT(posts.id) AS post_count
FROM users
LEFT JOIN posts ON posts.user_id = users.id
GROUP BY users.id;
Mobile analogy: A JOIN is like a Core Data relationship fetch — user.posts — except you write it explicitly in SQL and control exactly what columns come back.

Indexes — making queries fast

-- Without index: full table scan O(n) — slow at millions of rows
SELECT * FROM users WHERE email = 'g@example.com';

-- Add index on columns you query frequently
CREATE INDEX idx_users_email ON users(email);
-- Now the same query is O(log n)

-- Index columns used in: WHERE, JOIN ON, ORDER BY
-- Primary keys are indexed automatically

4Prisma ORM — type-safe database access

// schema.prisma
model User {
  id    Int    @id @default(autoincrement())
  name  String
  email String @unique
  posts Post[]
}
model Post {
  id     Int    @id @default(autoincrement())
  title  String
  user   User   @relation(fields: [userId], references: [id])
  userId Int
}
const prisma = new PrismaClient();

// Auto-completed, type-safe queries
const user = await prisma.user.findUnique({
  where:   { id: 1 },
  include: { posts: true }   // JOIN equivalent
});

await prisma.user.create({ data: { name: 'Gokul', email: 'g@x.com' } });
await prisma.user.update({ where: { id: 1 }, data: { name: 'Gokul P' } });
await prisma.user.delete({ where: { id: 1 } });
Mobile analogy: Prisma is like a type-safe API client generated from a schema — similar to Codable + URLSession, but for your database. You define the model, Prisma handles the SQL.

5Migrations — versioning your schema

# Generate and apply a migration (Prisma)
npx prisma migrate dev --name add_avatar_to_users
# Prisma generates: ALTER TABLE users ADD COLUMN avatar_url VARCHAR(500);
# Applies it to dev DB, records it in migrations/ folder

# Deploy to production
npx prisma migrate deploy
# Applies only un-run migrations, in order
Mobile analogy: Migrations are Core Data lightweight migrations — versioned schema changes that apply in sequence so every environment (dev, staging, prod) stays in sync.
