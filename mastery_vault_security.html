2Never store plaintext passwords

Passwords are hashed (one-way) with a slow, salted algorithm — bcrypt, scrypt, or Argon2. You never "decrypt" a password; you hash the login attempt and compare. Below is the browser-native SubtleCrypto to feel hashing (production uses bcrypt/Argon2, which add a per-user salt + cost factor).

JavaScript · runnable
// Same input -> same hash; one char changes everything
async function sha256(text) {
  const buf = await crypto.subtle.digest(
    'SHA-256', new TextEncoder().encode(text));
  return [...new Uint8Array(buf)]
    .map(b => b.toString(16).padStart(2, '0')).join('');
}
console.log('hunter2  →', await sha256('hunter2'));
console.log('hunter3  →', await sha256('hunter3'));
console.log('hunter2  →', await sha256('hunter2'), '(stable)');
Don't use raw SHA-256 for passwords. It's too fast — attackers brute-force billions/sec. Use bcrypt / Argon2, which are deliberately slow and salted. The demo above is to see hashing, not to ship it.

3Timing-safe comparison

Comparing secrets with === leaks information: it returns faster when the first characters differ. Attackers measure that to guess a token byte-by-byte. Always compare in constant time.

JavaScript · runnable
// Constant-time compare: time doesn't depend on where they differ
function safeEqual(a, b) {
  if (a.length !== b.length) return false;
  let diff = 0;
  for (let i = 0; i < a.length; i++)
    diff |= a.charCodeAt(i) ^ b.charCodeAt(i); // XOR, never early-return
  return diff === 0;
}
console.log('match: ', safeEqual('token_abc', 'token_abc'));
console.log('differ:', safeEqual('token_abc', 'token_xyz'));

4What's actually inside a JWT

Phase 1 issued JWTs. Here's the anatomy: three base64url parts — header.payload.signature. The payload is readable by anyone (not encrypted), only tamper-proof via the signature. Decode one live:

JavaScript · runnable
// Decode (NOT verify) a JWT's payload
const jwt = 'eyJhbGciOiJIUzI1NiJ9' +
  '.eyJ1c2VySWQiOjQyLCJyb2xlIjoiYWRtaW4ifQ' +
  '.s1gn4tur3';
const [header, payload] = jwt.split('.')
  .slice(0, 2)
  .map(p => JSON.parse(atob(p)));
console.log('header: ', header);
console.log('payload:', payload);
console.log('⚠ payload is visible — never put secrets in a JWT');

5Secrets management & least privilege

  • Never commit secrets. Use env vars locally, a secrets manager (AWS Secrets Manager, Vault, Doppler) in prod.
  • Rotate keys and tokens on a schedule and immediately after any suspected leak.
  • Least privilege (ties back to Phase 5 IAM): every service/key gets the minimum access it needs, nothing more.
  • Scan dependencies — npm audit, Dependabot, Trivy for images.
Shell · copy
# the four commands that catch most low-hanging fruit
npm audit --production          # known CVEs in deps
npx gitleaks detect             # secrets accidentally committed
docker scout cves my-image:tag  # vulnerabilities in your image
npm outdated                    # how far behind you are
Try it: Harden the Phase 1 API — (1) add an ownership check so a user can only edit their own tasks, (2) swap any plaintext password handling for bcrypt, (3) run npm audit and fix what it finds. Those three cover the most-exploited OWASP categories.
