🗼 The Watchtower

You can't operate what you can't see. When production breaks at 2am, observability is the difference between a five-minute fix and a five-hour outage.

1The three pillars

Observability is built from three signal types. Logs tell you what happened, metrics tell you how much / how often, traces tell you where the time went across services.

PillarAnswersMobile analogy
Logs"What exactly happened on this request?"Xcode console / os_log
Metrics"How many requests/sec? p99 latency?"MetricKit / crash-free rate
Traces"Which service made this slow?"Instruments time profiler

2Structured logging

Stop logging strings. Log JSON objects with consistent fields — they're searchable, filterable, and aggregatable. A requestId ties every line of one request together.

JavaScript · runnable
// A tiny structured logger — run it
const log = (level, msg, meta = {}) => {
  console.log(JSON.stringify({
    ts: new Date().toISOString(),
    level, msg, ...meta
  }));
};

const requestId = 'req_8f2a';
log('info', 'request received', { requestId, route: '/users/42' });
log('warn', 'slow query', { requestId, ms: 812 });
log('error', 'db timeout', { requestId, code: 'ETIMEDOUT' });
Rule: never log secrets, tokens, passwords, or full PII. Log a user id, never their email/password.

3Metrics — RED & USE

Two cheat-sheets tell you what to measure. RED for services you serve, USE for resources you consume.

  • RED — Rate (req/s), Errors (failed req/s), Duration (latency distribution)
  • USE — Utilization (% busy), Saturation (queue depth), Errors (faults)
Averages lie. A 200ms average latency can hide that 1% of users wait 4 seconds. Always look at percentiles — p50, p95, p99.

4Why percentiles beat averages

Run this to feel it: most requests are fast, a few are terrible. The average looks fine; the p99 tells the truth your users actually experience.

JavaScript · runnable
// 990 fast requests + 10 awful ones
const latencies = [
  ...Array(990).fill(0).map(() => 40 + Math.random() * 60),
  ...Array(10).fill(3500)
];

const pct = (arr, p) => {
  const s = [...arr].sort((a, b) => a - b);
  return s[Math.floor((p / 100) * s.length)];
};
const avg = latencies.reduce((a, b) => a + b) / latencies.length;

console.log('average:', Math.round(avg), 'ms — looks fine');
console.log('p50:', Math.round(pct(latencies, 50)), 'ms');
console.log('p95:', Math.round(pct(latencies, 95)), 'ms');
console.log('p99:', Math.round(pct(latencies, 99)), 'ms — the truth');

5Distributed tracing

One user action can touch five services. A trace stitches them together with a shared traceId; each hop is a span with its own timing. This is how you find which service ate the latency.

JavaScript · runnable
// Minimal span timer — propagate one traceId
const traceId = 'trace_a1b2';
const spans = [];
async function span(name, work) {
  const start = Date.now();
  await work();
  spans.push({ traceId, name, ms: Date.now() - start });
}
const sleep = ms => new Promise(r => setTimeout(r, ms));

await span('auth',    () => sleep(20));
await span('db.query',() => sleep(140));  // the culprit
await span('render',  () => sleep(15));
console.table(spans);

6SLOs & error budgets

A Service Level Objective is a target like "99.9% of requests succeed". Its inverse is your error budget — the allowed failure you can "spend" on risk. Burn the budget → freeze features, fix reliability.

JavaScript · runnable
// How much downtime does each "nines" SLO allow per month?
const minutesInMonth = 30 * 24 * 60;
[99, 99.9, 99.99].forEach(slo => {
  const budget = minutesInMonth * (1 - slo / 100);
  console.log(slo + '% → ' + budget.toFixed(1) + ' min downtime/month');
});
Try it: Add structured logging (with a per-request requestId) to the Express API from Phase 1. Then expose a /metrics endpoint counting requests and errors. You now have the two cheapest, highest-value observability wins.

The tools (for real systems)

Logs → Loki / ELK / CloudWatch Logs. Metrics → Prometheus + Grafana. Traces → OpenTelemetry + Jaeger / Tempo. The vendor-neutral standard to learn is OpenTelemetry — instrument once, export anywhere.
