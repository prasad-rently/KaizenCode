🧭 The Compass

The first six phases make you a strong builder. This is the other half of "Mobile Engineer → CTO": the judgement, people, and process that turn engineering into leadership.

1System design interviews (and meetings)

The same skill wins design interviews and real architecture reviews. Drive every discussion in this order — never jump to a database choice first.

  1. Clarify — functional + non-functional requirements. "How many users? Read- or write-heavy? Consistency vs availability?"
  2. Estimate — back-of-envelope: QPS, storage/day, bandwidth.
  3. Draw the boxes — client → LB → service → DB/cache → queue.
  4. Find the bottleneck — then scale that (cache, shard, queue).
  5. Trade-offs out loud — name what you're giving up. There's no perfect design.

2Capacity estimation — do the napkin math

Leaders quantify before they architect. Run a real back-of-envelope for a feed service.

JavaScript · runnable
// "Design a feed for 10M daily users"
const dau = 10_000_000;
const reqPerUser = 20;                 // feed opens/day
const seconds = 86_400;

const avgQps = (dau * reqPerUser) / seconds;
const peakQps = avgQps * 3;             // peak ≈ 3× average

console.log('avg  QPS:', Math.round(avgQps));
console.log('peak QPS:', Math.round(peakQps));

const perServer = 2000;               // QPS one node handles
console.log('servers needed at peak:', Math.ceil(peakQps / perServer));
// → now you know if this is a 3-box or 300-box problem

3Decisions: RFCs & ADRs

Leaders write so teams can align asynchronously and remember why. Two lightweight formats do most of the work.

DocPurposeWhen
RFC (Request for Comments)Propose a change, gather feedback before buildingBefore a big project
ADR (Architecture Decision Record)Record a decision + context + consequencesWhen you commit to a choice
Markdown · ADR template · copy
# ADR 007: Use PostgreSQL over MongoDB for billing

## Status
Accepted — 2026-06-30

## Context
Billing needs multi-row transactions and strong consistency.
Write volume is moderate; relationships are rich.

## Decision
Use PostgreSQL (CP system) as the system of record for billing.

## Consequences
+ ACID transactions, mature tooling, easy joins
- Vertical scaling ceiling; revisit if write volume 10x's
- Team must learn schema migrations (see Phase 2)
Why this matters: "we chose X because Y" written down kills the same debate being re-litigated every six months. ADRs are how a team remembers its own reasoning.

4Team topologies & ownership

  • Conway's Law — your system architecture will mirror your team structure. Design teams deliberately, because they shape the software.
  • Stream-aligned teams own a slice end-to-end (build and run it — "you build it, you run it").
  • Clear ownership — every service has one owning team. Shared ownership = no ownership.
  • Cognitive load — a team can only hold so much. Microservices (Phase 6) are as much about team boundaries as tech.

5On-call & incident response

Reliability is a leadership responsibility. A healthy culture is blameless — you fix the system, not the person.

  • Runbooks — documented "if this alert, do that" steps (pairs with Phase 6 Watchtower signals).
  • Sev levels — agree what Sev1/2/3 mean and who's paged for each.
  • Blameless postmortems — timeline, root cause, action items. The goal is a systemic fix, never a scapegoat.
  • Error budgets (from The Watchtower) decide when to slow features and invest in reliability.

6Strategy: build vs buy, and the CTO lens

The most senior question isn't "how do I build this?" but "should we, and is this the real problem?"

Build-vs-buy heuristic: build what is your core differentiator; buy/borrow everything else (auth, payments, email, search). Engineering time is your scarcest resource — spend it where it makes you unique.
The CTO architecture lens (carried from Phase 6): Where's the real bottleneck? What happens when a component dies? Are we solving a problem we have or one we imagine? Could boring tech (Postgres + Redis) solve this? What's the operational burden on a small team? Rule of three: add complexity only after hitting the same limit three times.

Keep going

Leadership compounds like code: small, consistent reps. Keep a decision journal, write one ADR for your next real choice, and run one blameless retro. That's Kaizen applied to how you lead, not just what you build.
