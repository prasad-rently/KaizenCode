🎫 The Gatekeeper

Who are you, and what are you allowed to do? Authentication and authorization are the front door of every system — get them wrong and nothing else matters. This goes deeper than The Vault: OAuth2, OIDC, PKCE, sessions vs tokens, SSO, and passkeys.

A senior/staff-level extension of The Vault. Auth leans heavily on the Cryptography topic — JWTs are signed (HMAC or signatures), PKCE uses hashing, passkeys are public-key signatures. If a primitive here is fuzzy, its crypto module explains it.

1AuthN vs AuthZ

Two different questions, constantly confused:

  • Authentication (AuthN) — who are you? Proving identity (password, passkey, an identity token).
  • Authorization (AuthZ) — what may you do? Granting access to resources (scopes, roles, permissions).

OAuth2 is an authorization framework; OIDC adds authentication on top. Mixing them up is the root of many auth bugs — "logging in with OAuth2 alone" (without OIDC) is a classic mistake.

Mobile and API backends lean on tokens; the trade-off is revocation, which is why access tokens are kept short-lived and paired with refresh tokens (§6).

3Anatomy of a JWT

A JWT is three base64url parts — header.payload.signature. The payload is readable by anyone (base64, not encryption); the signature only proves it wasn't tampered with. Sign one, read it, then try to forge it:

JavaScript · runnable (WebCrypto)
// A JWT = base64url(header).base64url(payload).base64url(signature)
const TE = new TextEncoder();
const b64u = buf => btoa(String.fromCharCode.apply(null, new Uint8Array(buf))).replace(/\+/g, '-').replace(/\//g, '_').replace(/=+$/, '');
const b64uStr = s => btoa(s).replace(/\+/g, '-').replace(/\//g, '_').replace(/=+$/, '');
async function hs256(secret, data) {
  const k = await crypto.subtle.importKey('raw', TE.encode(secret), { name: 'HMAC', hash: 'SHA-256' }, false, ['sign']);
  return b64u(await crypto.subtle.sign('HMAC', k, TE.encode(data)));
}

const header = b64uStr(JSON.stringify({ alg: 'HS256', typ: 'JWT' }));
const payload = b64uStr(JSON.stringify({ sub: '42', role: 'user' }));
const input = header + '.' + payload;
const sig = await hs256('server-secret', input);
console.log('token:', input + '.' + sig);

// anyone can DECODE the payload — it is not encrypted
console.log('decoded payload (public!):', JSON.parse(atob(payload.replace(/-/g, '+').replace(/_/g, '/'))));

// but forging it needs the secret: tamper role → user becomes admin?
const forged = header + '.' + b64uStr(JSON.stringify({ sub: '42', role: 'admin' }));
console.log('tampered token, old signature still valid?', (await hs256('server-secret', forged)) === sig);
Never put secrets in a JWT — the payload is public. And always verify the signature and the claims (exp, iss, aud) server-side. The infamous alg: none attack works when a server trusts the token's own header about how to verify it — pin the algorithm yourself.

4OAuth2 — delegated authorization

OAuth2 lets an app act on a user's behalf without handling their password. Four roles: the resource owner (user), the client (your app), the authorization server (issues tokens), and the resource server (the API). The modern default is the Authorization Code flow:

   App ──▶ /authorize ──▶ user logs in & consents
        ◀── redirect back with a short-lived CODE
   App ──▶ /token (code) ──▶ access_token (+ refresh_token)
   App ──▶ API with  Authorization: Bearer <access_token>

The old Implicit flow (tokens straight in the redirect URL) is deprecated — it leaked tokens through browser history and referrers. Today, everyone uses Authorization Code + PKCE, including single-page and mobile apps.

5PKCE — securing the code flow for public clients

A mobile or SPA client can't keep a client secret (it ships in the app bundle). PKCE ("pixy", Proof Key for Code Exchange) fixes that: the app invents a random code_verifier, sends only its hash (code_challenge = S256(verifier)) when starting the flow, and reveals the verifier only at token exchange. A stolen authorization code is useless without the verifier. See it live:

JavaScript · runnable (WebCrypto)
// PKCE: bind the authorization code to the app that started the flow
const TE = new TextEncoder();
const b64u = buf => btoa(String.fromCharCode.apply(null, new Uint8Array(buf))).replace(/\+/g, '-').replace(/\//g, '_').replace(/=+$/, '');
const s256 = async s => b64u(await crypto.subtle.digest('SHA-256', TE.encode(s)));

// 1) app generates a random secret verifier + its S256 challenge
const verifier = b64u(crypto.getRandomValues(new Uint8Array(32)));
const challenge = await s256(verifier);
console.log('code_verifier  (secret, on device):', verifier);
console.log('code_challenge (sent to /authorize):', challenge);

// 2) at /token the server recomputes S256(verifier) and compares to the challenge
console.log('server: S256(verifier) === challenge →', (await s256(verifier)) === challenge);

// 3) attacker steals the code but not the verifier → cannot match
console.log('attacker without verifier →', (await s256('wrong-guess')) === challenge);

6OIDC, refresh tokens & rotation

OIDC (OpenID Connect) is the thin identity layer on top of OAuth2. Alongside the access token it returns an id_token — a JWT about the user (sub, email, name) that your app verifies to log them in. Rule of thumb: access token = what you can do; id_token = who you are.

Refresh tokens solve the revocation trade-off: access tokens live minutes, and a longer-lived refresh token silently mints new ones. Best practice is refresh-token rotation — each use issues a new refresh token and invalidates the old one; if an old one is ever replayed, the server detects the reuse and revokes the whole family (a sign it was stolen).

7SSO — SAML & SCIM

  • SAML — the older XML-based enterprise SSO standard; still ubiquitous in big-company "Login with Okta/Azure AD". OIDC is its modern, JSON/JWT successor.
  • SCIM — the provisioning counterpart: it syncs user lifecycle (create/update/deactivate) from the company directory into your app, so deprovisioning an employee actually removes their access.

8Passkeys / WebAuthn — the phishing-resistant future

Passkeys replace passwords with a public-key key pair: your device keeps the private key (in the Secure Enclave / StrongBox — see the device-key idea), the server stores only the public key. Login is a signature over a server challenge. There's no shared secret to steal, and it's phishing-resistant because the signature is bound to the real site's origin — a fake site can't elicit a usable one. For a mobile engineer this is the direction of travel: Sign in with Apple/Google and platform passkeys already ride these rails.

Token storage on mobile & web. Never store tokens in plain localStorage (XSS reads it). On the web prefer HttpOnly, Secure, SameSite cookies; on mobile use the Keychain (iOS) / Keystore (Android). Always: Authorization Code + PKCE (never implicit), validate signature + exp + aud + iss, pin the alg, and keep access tokens short-lived.
