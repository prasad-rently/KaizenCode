🧰 The Foundry

The handful of data structures that show up again and again in real systems and design interviews. Not leetcode for its own sake — these are the exact tools behind caches, rate limiters, and sharding (Phases 5–6, The Edge).

1LRU cache — evict the Least Recently Used

A cache has finite memory, so when it fills you must evict something. LRU drops whatever hasn't been touched longest. A JS Map preserves insertion order, which makes this beautifully short:

JavaScript · runnable
class LRU {
  constructor(cap) { this.cap = cap; this.m = new Map(); }
  get(k) {
    if (!this.m.has(k)) return null;
    const v = this.m.get(k);
    this.m.delete(k); this.m.set(k, v);  // mark most-recent
    return v;
  }
  set(k, v) {
    if (this.m.has(k)) this.m.delete(k);
    this.m.set(k, v);
    if (this.m.size > this.cap)
      this.m.delete(this.m.keys().next().value); // oldest
  }
}
const c = new LRU(2);
c.set('a', 1); c.set('b', 2); c.get('a'); // touch 'a' → 'b' now oldest
c.set('c', 3);                     // evicts 'b'
console.log('b evicted?', c.get('b') === null, '| a kept?', c.get('a') === 1);
This is Redis' allkeys-lru policy (Phase 6 / The Edge). Now you know exactly what it does when memory fills.

2Bloom filter — "definitely no, probably yes"

Need to check membership in a huge set using tiny memory? A Bloom filter hashes each item to a few bits. It can give false positives but never false negatives — perfect for "have we seen this URL / is this username taken?" before hitting the DB.

JavaScript · runnable
class Bloom {
  constructor(size = 64) { this.size = size; this.bits = new Uint8Array(size); }
  hashes(s) {
    let h1 = 0, h2 = 0;
    for (const ch of s) { h1 = (h1 * 31 + ch.charCodeAt(0)) >>> 0;
                            h2 = (h2 * 57 + ch.charCodeAt(0)) >>> 0; }
    return [h1 % this.size, h2 % this.size];
  }
  add(s) { for (const i of this.hashes(s)) this.bits[i] = 1; }
  has(s) { return this.hashes(s).every(i => this.bits[i]); }
}
const b = new Bloom();
['gokul', 'admin', 'root'].forEach(u => b.add(u));
console.log("'admin' seen? ", b.has('admin')); // true (correct)
console.log("'alice' seen? ", b.has('alice')); // false (or rare false positive)

3Consistent hashing — shard without reshuffling

Sharding by hash(key) % N (Phase 6) breaks when N changes: adding one server remaps almost every key. Consistent hashing places servers on a ring so adding/removing a node moves only a small slice of keys. Prove it:

JavaScript · runnable
const hash = s => { let h = 0; for (const c of s) h = (h * 31 + c.charCodeAt(0)) >>> 0; return h % 360; };
function place(keys, nodes) {                 // nearest node clockwise on a 0-360 ring
  const ring = nodes.map(n => ({ n, pos: hash(n) })).sort((a, b) => a.pos - b.pos);
  const map = {};
  for (const k of keys) {
    const h = hash(k);
    map[k] = (ring.find(r => r.pos >= h) || ring[0]).n;
  }
  return map;
}
const keys = ['user:42', 'cart:7', 'order:1093', 'photo:x9', 'token:abcd', 'post:557', 'job:queue', 'sess:kp3'];
const nodes = ['cache-tokyo', 'cache-paris', 'cache-ohio'];
const before = place(keys, nodes);
const after  = place(keys, [...nodes, 'cache-mumbai']); // add a 4th node
const moved = keys.filter(k => before[k] !== after[k]);
console.log('keys that moved when adding a node:', moved);
console.log('only', moved.length, 'of', keys.length, 'remapped (mod-N would move ~all)');

4Sliding-window rate limiter

The Gateway showed a token bucket. Here's the other classic: a sliding window that counts requests in the last T ms and rejects over the cap — smoother than fixed windows at the boundary.

JavaScript · runnable
function limiter(max, windowMs) {
  let hits = [];
  return (now) => {
    hits = hits.filter(t => now - t < windowMs); // drop old
    if (hits.length >= max) return '429 blocked';
    hits.push(now);
    return '200 ok (' + hits.length + '/' + max + ')';
  };
}
const allow = limiter(3, 1000);   // 3 per second
let t = 0;
for (const dt of [0, 100, 100, 100, 900]) { t += dt; console.log('t=' + t + 'ms → ' + allow(t)); }

5Counting the huge: HyperLogLog

How do you count unique visitors across a billion events without storing every id? HyperLogLog estimates cardinality using tiny memory (KBs) by tracking the longest run of leading zeros in item hashes — more leading zeros seen ⇒ more distinct items. Redis' PFCOUNT is this.

The theme of this module: trade a little accuracy or a little structure for enormous savings in time or memory. Exactness is often the expensive luxury you don't need at scale.
