🌊 The Streaming Mind

Phase 6 introduced Kafka. This module rewires how you think about time: from request/response snapshots to a continuous stream of events — the model behind real-time apps, audit logs, and undo.

1Request/response vs event-driven

In request/response the caller waits for an answer. In event-driven, a producer emits a fact ("OrderPlaced") and walks away; any number of consumers react independently. Loose coupling is the whole point.

Mobile analogy: request/response is a network GET you await. Event-driven is NotificationCenter / Combine publishers — post once, every subscriber reacts.

Build a tiny in-memory pub/sub — the heart of every event system. One publish, many subscribers react:

JavaScript · runnable
// Minimal event bus (pub/sub)
function createBus() {
  const subs = {};
  return {
    on: (topic, fn) => (subs[topic] ??= []).push(fn),
    emit: (topic, data) => (subs[topic] || []).forEach(fn => fn(data)),
  };
}
const bus = createBus();
bus.on('OrderPlaced', o => console.log('📧 email receipt for #' + o.id));
bus.on('OrderPlaced', o => console.log('📦 notify warehouse for #' + o.id));
bus.on('OrderPlaced', o => console.log('📊 update analytics: $' + o.total));

bus.emit('OrderPlaced', { id: 1001, total: 49 }); // 1 event, 3 reactions

3Event sourcing — state is a fold over events

Instead of storing current state, store the immutable log of events that produced it. Current state is just a replay. You get a perfect audit trail and time-travel for free.

JavaScript · runnable
// A bank account as an event log (never store the balance directly)
const events = [
  { type: 'opened',     amount: 0 },
  { type: 'deposited',  amount: 100 },
  { type: 'withdrew',   amount: 30 },
  { type: 'deposited',  amount: 50 },
];
// "fold" the log into current state
const balance = events.reduce((bal, e) => {
  if (e.type === 'deposited') return bal + e.amount;
  if (e.type === 'withdrew')  return bal - e.amount;
  return bal;
}, 0);
console.log('balance:', balance);
console.log('state at event 2:', events.slice(0,2).reduce((b,e)=>e.type==='deposited'?b+e.amount:b,0));
This is Kafka's superpower (Phase 6): because events are retained, a brand-new service can replay history and build its own view — no migration, no backfill scripts.

4CQRS — split reads from writes

Command Query Responsibility Segregation: the write model (commands, normalized, consistent) is separate from the read model (denormalized, fast, query-shaped). Events keep them in sync.

JavaScript · runnable
// Write side emits events; read side projects them into a fast view
const readModel = {};                       // optimised for queries
function project(event) {                    // a "projection"
  if (event.type === 'UserRegistered')
    readModel[event.id] = { id: event.id, name: event.name, posts: 0 };
  if (event.type === 'PostCreated')
    readModel[event.userId].posts++;          // pre-computed count
}
[ { type:'UserRegistered', id:1, name:'Gokul' },
  { type:'PostCreated', userId:1 },
  { type:'PostCreated', userId:1 } ].forEach(project);

console.log(readModel[1]); // read needs no JOIN — count is ready

5Delivery guarantees & idempotency

Distributed messaging is "at-least-once" in practice — duplicates happen. The fix isn't exactly-once delivery (nearly impossible); it's making consumers idempotent so processing twice is harmless.

JavaScript · runnable
// Dedupe by event id → safe to redeliver
const processed = new Set();
function handle(event) {
  if (processed.has(event.id)) return console.log('⏩ skip duplicate ' + event.id);
  processed.add(event.id);
  console.log('✅ charged card for ' + event.id);
}
// the broker redelivers #777 — but the card is charged ONCE
['777', '778', '777'].forEach(id => handle({ id }));
Try it: Add a WebSocket (or SSE) endpoint to the Phase 3 app so new items appear live without refresh. Then model one action (e.g. "task completed") as an event and store the log — rebuild the task list by folding events instead of mutating rows.
