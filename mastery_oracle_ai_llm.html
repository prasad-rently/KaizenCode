🔮 The Oracle

The skill of the moment: wiring large language models into real products. You already call REST APIs (Phase 1) — an LLM is one more, except its output is probabilistic, not deterministic. That one difference changes everything.

1An LLM is a probabilistic function

Same input, possibly different output. It predicts the next token over and over. Two consequences you must design around: you can't unit-test exact strings, and temperature tunes creativity vs determinism.

JavaScript · runnable
// Toy next-token sampler: higher temperature = more random
function sample(logits, temp) {
  const exps = logits.map(l => Math.exp(l / temp));
  const sum = exps.reduce((a, b) => a + b, 0);
  return exps.map(e => (e / sum * 100).toFixed(1) + '%');
}
const logits = [2.0, 1.0, 0.2]; // scores for 3 candidate tokens
console.log('temp 0.2 (focused): ', sample(logits, 0.2));
console.log('temp 1.0 (balanced):', sample(logits, 1.0));
console.log('temp 2.0 (wild):    ', sample(logits, 2.0));

2Tokens & the context window

Models read tokens (~¾ of a word in English), not characters. Every model has a fixed context window — prompt + response must fit. You pay per token, so tokens are literally your budget.

JavaScript · runnable
// Rough token estimate (real tokenizers are model-specific)
const estTokens = text => Math.ceil(text.length / 4);
const prompt = 'Summarise the quarterly report for the board in 3 bullets.';
const window = 200000;               // e.g. a large context window

const used = estTokens(prompt);
console.log('prompt ≈', used, 'tokens');
console.log('room left ≈', window - used, 'tokens for context + reply');
console.log('cost scales with tokens → trim prompts, cache system prompts');

3Calling a model (Claude Messages API)

The shape is a plain HTTPS POST — identical mental model to any API you've consumed. Default to the latest, most capable models (e.g. Claude Opus 4.8 / Sonnet 5) and keep the key server-side (The Vault).

JavaScript · copy
const res = await fetch('https://api.anthropic.com/v1/messages', {
  method: 'POST',
  headers: {
    'x-api-key': process.env.ANTHROPIC_API_KEY,   // never in client code
    'anthropic-version': '2023-06-01',
    'content-type': 'application/json',
  },
  body: JSON.stringify({
    model: 'claude-sonnet-5',
    max_tokens: 1024,
    system: 'You are a concise financial analyst.',
    messages: [{ role: 'user', content: 'Summarise Q3 in 3 bullets.' }],
  }),
});
const data = await res.json();
console.log(data.content[0].text);

4Embeddings & semantic search

An embedding turns text into a vector so that similar meaning → nearby vectors. Compare with cosine similarity. This is the engine of search, recommendations, and RAG.

JavaScript · runnable
// Cosine similarity — how "aligned" two vectors are (1 = identical)
const cosine = (a, b) => {
  const dot = a.reduce((s, x, i) => s + x * b[i], 0);
  const mag = v => Math.sqrt(v.reduce((s, x) => s + x * x, 0));
  return (dot / (mag(a) * mag(b))).toFixed(3);
};
// pretend these came from an embedding model
const query    = [0.9, 0.1, 0.0];  // "how do I reset my password"
const passwordDoc = [0.8, 0.2, 0.1];
const billingDoc  = [0.1, 0.2, 0.9];
console.log('query ↔ password doc:', cosine(query, passwordDoc)); // high
console.log('query ↔ billing doc: ', cosine(query, billingDoc));  // low

5RAG — Retrieval-Augmented Generation

Models don't know your private data and can hallucinate. RAG fixes both: retrieve relevant chunks from your own corpus, then stuff them into the prompt as grounding. Build the retrieval core:

JavaScript · runnable
// A tiny RAG retriever: embed corpus, score vs query, take top-k
const cos = (a, b) => {
  const d = a.reduce((s, x, i) => s + x * b[i], 0);
  const m = v => Math.sqrt(v.reduce((s, x) => s + x * x, 0));
  return d / (m(a) * m(b));
};
const corpus = [
  { text: 'Refunds are processed in 5-7 business days.', vec: [0.1, 0.9, 0.1] },
  { text: 'Reset your password from Settings > Security.', vec: [0.9, 0.1, 0.1] },
  { text: 'Our office is open 9am to 6pm IST.',          vec: [0.1, 0.1, 0.9] },
];
const queryVec = [0.85, 0.15, 0.05];      // "I forgot my password"
const topK = corpus
  .map(c => ({ text: c.text, score: +cos(queryVec, c.vec).toFixed(2) }))
  .sort((a, b) => b.score - a.score)
  .slice(0, 1);
console.log('retrieved context:', topK);
console.log('→ inject that text into the prompt so the model answers from facts');
The RAG pipeline: chunk documents → embed each chunk → store vectors (pgvector, Pinecone, Weaviate) → at query time embed the question, retrieve top-k, and prepend them to the prompt. It's Phase 2 (a database) + section 4 (embeddings) combined.

6Prompt engineering & structured output

  • Be specific — role, task, constraints, and the exact output format.
  • Few-shot — show 1–3 examples of input→output; models imitate patterns.
  • Ask for JSON — then JSON.parse it, so the LLM plugs into normal code.
JavaScript · runnable
// Treat the model like a typed function: prompt in, JSON out, parse, use
const promptTemplate = (review) => `Classify the sentiment of the review.
Return ONLY JSON: {"sentiment":"positive|negative|neutral","score":0-1}
Review: "${review}"`;

console.log(promptTemplate('The app is fast but crashes on launch.'));
// Model would reply with JSON; your code then does:
const fakeModelReply = '{"sentiment":"negative","score":0.7}';
const parsed = JSON.parse(fakeModelReply);
console.log('parsed →', parsed.sentiment, parsed.score);

7Agents & tool use

An agent is an LLM in a loop that can call tools (functions) to act, not just talk. You describe tools; the model decides which to call and with what arguments; you execute and feed the result back. Simulate the router:

JavaScript · runnable
// Tools the agent may call
const tools = {
  getWeather: (city) => `${city}: 31°C, humid`,
  addNumbers: (a, b) => a + b,
};
// The model "decides" to emit a structured tool call (mocked here)
const toolCall = { name: 'getWeather', args: ['Chennai'] };

// Your runtime dispatches it and returns the observation to the model
const result = tools[toolCall.name](...toolCall.args);
console.log('tool:', toolCall.name, '→', result);
console.log('the observation is fed back; the loop continues until done');
Ship responsibly: validate/parse all model output (never eval it), set token & spend limits, add timeouts + retries (The Bulwark), log prompts/outputs for eval (The Watchtower), and guard against prompt injection when user text enters the prompt.
Try it: Add an /ask endpoint to the Phase 1 API that (1) embeds the question, (2) retrieves the top-k chunks from a small doc set, (3) calls a Claude model with those chunks as context, and (4) returns the answer. That's a complete RAG feature in ~40 lines.
