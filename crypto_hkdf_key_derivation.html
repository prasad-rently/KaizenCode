🧪 HKDF & PBKDF2 — Key Derivation

Raw key material — an ECDH shared secret, or a human password — is rarely a good key as-is. Key derivation functions refine it into clean, full-strength, purpose-specific keys. This is the quiet step that sits between "agree a secret" and "encrypt the data."

Part of the Cryptography topic. It's the missing middle of the ECDH → AES-GCM pipeline: ECDH agrees a shared secret, a KDF turns it into an AES key, AES-GCM encrypts. Two jobs, two tools — HKDF for high-entropy secrets, PBKDF2 (and friends) for passwords.

1Why you can't use raw key material directly

A key for AES-GCM must be a uniform, unpredictable block of bits. But the material you start with usually isn't:

  • An ECDH shared secret is a curve point — it's secret, but its bits aren't perfectly uniform, and it's a fixed size you may need to reshape.
  • A password is worse: short, low-entropy, and guessable. Using it directly as a key would be trivially brute-forced.

A KDF fixes this: it takes imperfect input and produces one or more keys that are indistinguishable from random — and, for passwords, deliberately expensive to compute.

2HKDF = Extract, then Expand

Analogy — an oil refinery. Crude oil (your raw secret) is useful but unrefined. HKDF works in two stages, just like a refinery:

  • Extract — condense the messy input (plus an optional salt) into one uniform, high-quality intermediate key, the PRK (pseudorandom key). This is "distil the crude into clean feedstock."
  • Expand — stretch the PRK into as many output keys as you need, each tagged with a distinct info label. This is "make gasoline, diesel, and jet fuel from the same feedstock."
   raw secret ─┐
   salt ───────┼─▶ [EXTRACT] ─▶ PRK ─▶ [EXPAND] ─┬─▶ key "encryption"
                                    (info label) ├─▶ key "authentication"
                                                 └─▶ key "cookie-signing"

The magic is that all those output keys are cryptographically independent — learning one tells you nothing about the others.

3See it: one secret → many independent keys

Below, a single shared secret produces two different 256-bit keys just by changing the info label — and re-deriving with the same inputs gives the identical key back. Run it:

JavaScript · runnable (WebCrypto)
// HKDF: turn ONE secret into MANY purpose-specific keys
const TE = new TextEncoder();
const hex = u => [...new Uint8Array(u)]
  .map(x => x.toString(16).padStart(2, '0')).join('');

// pretend this came out of an ECDH key agreement
const prk = await crypto.subtle.importKey(
  'raw', TE.encode('shared-secret-from-ECDH'), 'HKDF', false, ['deriveBits']);

const derive = (info) => crypto.subtle.deriveBits(
  { name: 'HKDF', hash: 'SHA-256', salt: TE.encode('app-salt'), info: TE.encode(info) },
  prk, 256);

const encKey = hex(await derive('app:encryption'));
const macKey = hex(await derive('app:authentication'));
console.log('encryption key   :', encKey.slice(0, 32) + '…');
console.log('authentication key:', macKey.slice(0, 32) + '…');
console.log('different keys from one secret?', encKey !== macKey);
console.log('re-derive encryption key → identical?', hex(await derive('app:encryption')) === encKey);

The info label gives you domain separation: the same secret safely powers many keys, each bound to a purpose, so a key meant for signing can never accidentally be the key used for encryption. The salt is not secret and may be empty, but a random salt strengthens the Extract step.

4Passwords are different — use PBKDF2 / scrypt / Argon2

HKDF assumes its input already has plenty of entropy. A password does not — so HKDF is the wrong tool for it. Password KDFs add one crucial property: they are deliberately slow (a tunable "work factor"), so that checking one guess is cheap for you but checking billions is ruinously expensive for an attacker with a stolen database.

JavaScript · runnable (WebCrypto)
// PBKDF2: stretch a low-entropy PASSWORD into a key — slowly, on purpose
const TE = new TextEncoder();
const hex = u => [...new Uint8Array(u)]
  .map(x => x.toString(16).padStart(2, '0')).join('');
const pw = await crypto.subtle.importKey(
  'raw', TE.encode('hunter2'), 'PBKDF2', false, ['deriveBits']);

const derive = (salt, iters) => crypto.subtle.deriveBits(
  { name: 'PBKDF2', hash: 'SHA-256', salt: TE.encode(salt), iterations: iters }, pw, 256);

const t0 = performance.now();
const k1 = hex(await derive('alice-salt', 600000));   // OWASP-recommended floor
const ms = Math.round(performance.now() - t0);
console.log('derived key:', k1.slice(0, 32) + '…');
console.log(600000 + ' iterations took ~' + ms + 'ms — the slowness IS the defense');
console.log('same password + salt → same key?', hex(await derive('alice-salt', 600000)) === k1);
console.log('different salt → different key?', hex(await derive('bob-salt', 600000)) !== k1);
Iterations and salts matter. Use a high iteration count (OWASP suggests ≥600,000 for PBKDF2-HMAC-SHA256, tuned upward over time) and a unique random salt per password. For new systems, Argon2id is the modern first choice — it also resists GPU/ASIC cracking via memory hardness. PBKDF2 is fine and FIPS-approved; scrypt and Argon2 are stronger against specialized hardware.

5HKDF vs PBKDF2 at a glance

6Where you'll meet key derivation

  • TLS 1.3 key schedule — HKDF derives every traffic key from the handshake secret.
  • ECIES / your ECDH module — the shared secret is run through HKDF to get the AES-GCM key.
  • Signal — the Double Ratchet chains keys forward with HKDF on every message.
  • Login systems — passwords are stored as PBKDF2/scrypt/Argon2 outputs, never as the password itself.
  • Password-based encryption — encrypting a file with a passphrase derives the key with PBKDF2/Argon2 first.
