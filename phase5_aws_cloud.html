Phase 5 · Weeks 17–20

Cloud: AWS

The infrastructure layer every CTO must own. You rent computers. Amazon manages the hardware.

1The mental model

AWS is the world's largest cloud provider — 200+ services, but you need only 8 to deploy most production apps. Instead of buying physical servers, you rent compute, storage, and networking by the second or GB.

Mobile analogy: Before iCloud, apps managed their own file servers. AWS is to infrastructure what iCloud is to storage — you consume it via API without worrying about hardware, cooling, or physical security.
Region
A geographic area: ap-south-1 (Mumbai), us-east-1 (Virginia). Pick the one closest to your users. Data doesn't automatically cross regions.
Availability Zone (AZ)
Independent data centers within a region. Deploy to 2+ AZs for high availability. One AZ failing shouldn't take down your app.

2EC2 — virtual machines

Elastic Compute Cloud gives you a virtual machine in the cloud. SSH in and run anything — Node.js, Python, Go, Docker. The closest thing to having your own server.

# After launching an EC2 instance (Ubuntu), SSH in:
ssh -i "~/.ssh/my-key.pem" ubuntu@ec2-54-123-45-67.compute.amazonaws.com

# Deploy your app:
git clone https://github.com/you/myapp && cd myapp
npm install && npm run build

# Keep it running with PM2 (process manager)
pm2 start dist/server.js --name api
pm2 startup  # auto-restart on server reboot
pm2 save
Key concepts
AMI: OS image template (like a Docker base image for a whole VM)
Security Group: Firewall rules — open port 443 (HTTPS), 22 (SSH)
Elastic IP: Static IP address that survives restarts
Key Pair: SSH keys to access your instance
Common instance types
t3.micro — free tier, dev/test
t3.small — small production
t3.medium — moderate load
m6i.large — general production
c6i.xlarge — compute-heavy

3S3 — object storage

Simple Storage Service stores any file — images, videos, PDFs, backups, static websites. Virtually unlimited capacity, 11 nines of durability, cheap.

const { S3Client, PutObjectCommand, GetObjectCommand } = require('@aws-sdk/client-s3');
const s3 = new S3Client({ region: 'ap-south-1' });

// Upload a file (e.g., user avatar)
await s3.send(new PutObjectCommand({
  Bucket:      'my-app-uploads',
  Key:         'avatars/user-42.jpg',  // the "path" inside the bucket
  Body:        fileBuffer,
  ContentType: 'image/jpeg',
}));

// Public URL (if bucket is configured for public access)
// https://my-app-uploads.s3.ap-south-1.amazonaws.com/avatars/user-42.jpg
Mobile analogy: S3 is like a global file server. You store the file URL in your database, load the image from S3 in your app. Your EC2 server never stores user uploads on its own disk — that would be lost on a redeployment.

4RDS — managed databases

RDS runs PostgreSQL/MySQL on AWS-managed hardware. AWS handles automated backups, OS patching, multi-AZ failover, and read replicas. You just connect with a connection string.

Why RDS over self-hosted
Automated daily backups
Point-in-time recovery (oops!)
Read replicas for scale
Multi-AZ automatic failover
No OS-level maintenance
RDS alternatives
Supabase: Postgres + auth + storage, free tier, great DX
PlanetScale: MySQL, branching model
Railway: Simple, cheap, great for side projects

5Lambda — serverless functions

Lambda runs your code in response to events — HTTP requests, S3 file uploads, scheduled timers — without you managing any server. Pay per invocation (first 1M free/month).

// Lambda handler — runs in response to an event
exports.handler = async (event) => {
  const { userId } = JSON.parse(event.body);
  const user = await db.findUser(userId);
  return {
    statusCode: 200,
    headers: { 'Content-Type': 'application/json' },
    body: JSON.stringify(user)
  };
};

// Common triggers:
// API Gateway → Lambda (HTTP API endpoint)
// S3 event    → Lambda (resize image on upload)
// EventBridge → Lambda (scheduled cron job)
// SNS/SQS     → Lambda (process queue messages)
Cold start: First invocation after idle period takes 100ms–3s. Keep latency-sensitive Lambdas warm with scheduled pings, or use Provisioned Concurrency.

6IAM — identity and access management

IAM controls who can do what to which AWS resources. Misconfigured IAM is the #1 cause of AWS data breaches. Learn it thoroughly.

// IAM Policy — allow Lambda to write to S3, nothing else
{
  "Version": "2012-10-17",
  "Statement": [{
    "Effect":   "Allow",
    "Action":   ["s3:PutObject", "s3:GetObject"],
    "Resource": "arn:aws:s3:::my-uploads/*"
  }]
}
Principle of least privilege: Every user, EC2 instance, and Lambda should have only the minimum permissions it needs. Never use root credentials in production. Never hardcode AWS keys in code.
  • User: A person with long-lived credentials (access key + secret)
  • Role: A set of permissions attached to an AWS service (EC2, Lambda)
  • Policy: JSON document defining allowed/denied actions on resources
  • Best practice: Use roles for services, never hardcode access keys in app code

7VPC — virtual private cloud

VPC is your private network on AWS. Resources are isolated from the internet unless explicitly exposed.

Public subnet
Has a route to the internet via Internet Gateway. Put your Load Balancer and NAT Gateway here. Direct internet access.
Private subnet
No inbound internet route. Put EC2 app servers and RDS databases here. Only reachable from within the VPC — safest for your backend.
Best practice: Your RDS database should always live in a private subnet. The internet cannot reach it directly — only your app server (also in the VPC) can connect to it.

8CloudWatch + Route 53

CloudWatch
Logs, metrics, and alarms for all AWS services. Set an alarm when CPU exceeds 80%, when error rate spikes, or when a Lambda fails. First stop when production breaks.
Route 53
AWS's DNS service. Register domains, create A records pointing to EC2 / Load Balancers. Health checks + failover routing built in.
Standard production architecture: Route 53 → CloudFront (CDN) → Application Load Balancer → EC2 in private subnet → RDS in private subnet. S3 for assets. CloudWatch for monitoring. IAM roles for all service-to-service auth. This pattern covers 90% of production web apps.
