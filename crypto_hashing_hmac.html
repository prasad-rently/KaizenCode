🧬 Hashing & HMAC — Digital Fingerprints

How to reduce any data — a word or a whole novel — to a short, unforgeable fingerprint that proves it wasn't tampered with, and how a secret key turns that fingerprint into proof of who produced it.

Part of the Cryptography topic. Where ECDH was about confidentiality (keeping data secret), hashing is about integrity (proving data is unchanged). Different goal, and one of the most-used primitives in all of computing.

1What a hash is

A cryptographic hash function (SHA-256, SHA-3, BLAKE3…) takes any input and returns a fixed-size output — always 64 hex characters for SHA-256, whether the input is one letter or a gigabyte.

Analogy — the blender. Throw fruit into a blender and you get a smoothie. Same fruit → same smoothie every time. Change one grape → a visibly different smoothie. And you can never un-blend the smoothie back into fruit. A hash is exactly that: deterministic, fixed-size, and one-way.

The four properties that make it cryptographic:

  • Deterministic — same input always gives the same digest.
  • One-way (pre-image resistance) — given a digest, you can't work backwards to the input.
  • Avalanche effect — change one bit of input and roughly half the output bits flip.
  • Collision resistance — you can't find two different inputs with the same digest.

2See it: determinism + the avalanche effect

Hash "hello", then "hellp" (one letter later in the alphabet). The two digests share almost nothing — yet hashing "hello" again returns the exact same value. Run it:

JavaScript · runnable (WebCrypto)
// SHA-256 in the browser — any input → 64 hex chars
async function sha256(text) {
  const data = new TextEncoder().encode(text);
  const buf = await crypto.subtle.digest('SHA-256', data);
  return [...new Uint8Array(buf)]
    .map(b => b.toString(16).padStart(2, '0')).join('');
}

const a = await sha256('hello');
const b = await sha256('hellp');   // one letter changed
console.log('hello →', a);
console.log('hellp →', b);
console.log('hello again →', await sha256('hello'), '(identical — deterministic)');

// how much of the output changed from a 1-letter tweak?
let diff = 0;
for (let i = 0; i < a.length; i++) if (a[i] !== b[i]) diff++;
console.log('hex chars different:', diff, 'of', a.length, '← avalanche effect');

3What plain hashing is good for

  • Integrity / download verification — a site publishes a file's SHA-256; you hash your download and compare. Match = not corrupted or tampered.
  • Content addressing — Git names every commit and file by its hash; change one byte and it's a different object. IPFS and Docker layers work the same way.
  • Deduplication — identical files hash identically, so you store the bytes once.
  • Commitment — publish a hash now to prove you knew something, reveal the input later.
Plain hashing is not for passwords. SHA-256 is too fast — an attacker tries billions of guesses per second against a stolen database. Passwords need a deliberately slow, salted function (bcrypt / scrypt / Argon2) — see §5 below.

4The SHA family — SHA-2, SHA-3, and hedging

SHA-256 belongs to the SHA-2 family (SHA-224/256/384/512) — the general-purpose standard behind file integrity, digital signatures, and content addressing (Git commit IDs, blockchains).

SHA-3 (Keccak) is not a patch to SHA-2 — it's a deliberately different internal design (a "sponge" construction instead of SHA-2's Merkle–Damgård). It was standardised as a hedge: if a structural weakness were ever found in SHA-2, the world would already have a mature alternative built on entirely different math. Both are current and secure — SHA-3 is an alternative, not a replacement.

5Password hashing — why "fast" is dangerous

Here's the counter-intuitive part: for passwords, a fast hash is a liability. SHA-256 is built to be blazing fast — exactly what an attacker with a stolen password database wants, because they can try enormous numbers of guesses. Measure your own browser (single-threaded, via Web Crypto):

The fix is a password hashing function that's deliberately slow and tunable:

  • Salt — a unique random value per password, stored next to the hash. It defeats precomputed "rainbow tables" and makes two users with the same password store different hashes. But salt alone doesn't stop a targeted brute-force — you also need slowness.
  • bcrypt / scrypt / Argon2 — built with a work factor you dial up over time so each single guess costs real CPU (and, for scrypt/Argon2, real memory). Argon2id is the modern recommendation: it's memory-hard, which blunts the GPU/ASIC rigs that would otherwise parallelise millions of guesses.
Rule: never store passwords with plain SHA-256/512. Use Argon2id (or bcrypt/scrypt) with a tuned work factor. The key-derivation module shows the same slow-KDF idea (PBKDF2) with a runnable demo.

6MD5 & SHA-1 — why they're dead

Collision resistance erodes as attacks improve. These two are broken and must never be used for security:

HashWhy it's deadNamed attack
MD5Practical collisions — two different inputs with the same hash — can be produced in seconds, so an MD5 "fingerprint" is forgeable. Acceptable only as a non-security checksum against accidental corruption.Collisions since 2004; a forged CA certificate in 2008.

Both are absent from Web Crypto's modern surface (SHA-1 lingers only as legacy). The lesson: a hash being "standard" is a snapshot, not a guarantee — which is why we follow NIST guidance and migrate off algorithms as attacks mature.

7HMAC — a fingerprint only a key-holder can make

A plain hash proves data is unchanged, but anyone can compute it — so anyone can change the data and recompute a matching hash. To prove a message came from someone who holds a shared secret, you use HMAC (Hash-based Message Authentication Code): a hash mixed with a secret key.

Analogy. A plain hash is a wax impression anyone can make. HMAC is a wax seal stamped with your signet ring — others can check it matches your ring, but can't produce it without the ring.

Don't roll your own. The naïve hash(key + message) is vulnerable to length-extension attacks. HMAC's specific two-pass construction (hash(key ⊕ opad ‖ hash(key ⊕ ipad ‖ msg))) exists precisely to prevent that. Always use the library's HMAC.

Below, one party signs a message with a shared key; the other verifies it. Then an attacker (who lacks the key) tampers the message — and verification fails:

JavaScript · runnable (WebCrypto)
// HMAC-SHA256: authenticated fingerprint, keyed by a shared secret
const enc = new TextEncoder();
const key = await crypto.subtle.importKey(
  'raw', enc.encode('shared-secret-key'),
  { name: 'HMAC', hash: 'SHA-256' }, false, ['sign', 'verify']);
const hex = u => [...new Uint8Array(u)]
  .map(x => x.toString(16).padStart(2, '0')).join('');

// Sender computes a tag over the message
const message = 'transfer $100 to Bob';
const tag = await crypto.subtle.sign('HMAC', key, enc.encode(message));
console.log('message:', message);
console.log('tag    :', hex(tag).slice(0, 32) + '…');

// Receiver verifies with the SAME key
const good = await crypto.subtle.verify('HMAC', key, tag, enc.encode(message));
console.log('verify original message →', good);

// Attacker changes "Bob" → "Eve" but can't forge a new tag (no key)
const forged = 'transfer $100 to Eve';
const bad = await crypto.subtle.verify('HMAC', key, tag, enc.encode(forged));
console.log('verify tampered message →', bad, '← rejected');

8Hashing vs HMAC vs encryption

Plain hashHMACEncryption
Needs a key?NoYes (shared secret)Yes
Reversible?NoNoYes (that's the point)
Proves integrity?YesYesOnly with an auth tag (e.g. GCM)
Proves who sent it?NoYes (any key-holder)No, by itself
Hides the content?NoNoYes

They solve different problems and are routinely combined — e.g. encrypt for secrecy, then HMAC (or use AEAD like AES-GCM) for integrity.

9Where you'll meet HMAC in the wild

  • JWTs (HS256) — the signature on a JSON Web Token is literally HMAC-SHA256 over the header + payload.
  • Webhook verification — Stripe, GitHub, Slack sign each webhook with HMAC so you can confirm it really came from them.
  • API request signing — AWS Signature v4 HMACs your request so AWS knows it wasn't altered in transit.
  • Cookie / session integrity — signed cookies use HMAC so a user can't tamper with their own session data.
Compare tags in constant time. Checking a received tag with a normal === can leak, via tiny timing differences, how many leading bytes matched — letting an attacker guess a valid tag byte by byte. Use a constant-time compare (Node's crypto.timingSafeEqual), or better, WebCrypto's verify() as above, which handles it for you.
