🤹 The Juggler

One thread, many balls in the air. Node runs your JavaScript on a single thread — mastering the event loop, promises, and concurrency limits is what separates code that scales from code that stalls.

1The event loop & execution order

Synchronous code runs first. Then microtasks (resolved promises), then macrotasks (setTimeout). Predict the output before you run it — this trips up almost everyone.

JavaScript · runnable
console.log('1 — sync');
setTimeout(() => console.log('4 — macrotask (setTimeout)'), 0);
Promise.resolve().then(() => console.log('3 — microtask (promise)'));
console.log('2 — sync');
// Order: 1, 2, 3, 4 — microtasks jump the setTimeout queue
Mobile analogy: the event loop is DispatchQueue.main. Blocking it (a heavy sync loop) freezes everything — the server's equivalent of a janky, frozen UI thread.

2Sequential vs parallel

Awaiting in a loop runs tasks one after another. Independent async work should run concurrently with Promise.all. The time difference is dramatic.

JavaScript · runnable
const task = (ms) => new Promise(r => setTimeout(() => r(ms), ms));

const t0 = Date.now();
// ❌ sequential: 100 + 100 + 100 ≈ 300ms
for (const ms of [100, 100, 100]) await task(ms);
console.log('sequential:', Date.now() - t0, 'ms');

const t1 = Date.now();
// ✅ parallel: all at once ≈ 100ms
await Promise.all([task(100), task(100), task(100)]);
console.log('parallel:  ', Date.now() - t1, 'ms');

3Bounded concurrency (a semaphore)

Promise.all over 10,000 items fires 10,000 requests at once — and melts the DB or gets you rate-limited (The Gateway). Cap it: run at most N in flight.

JavaScript · runnable
// Process items with at most `limit` running concurrently
async function pool(items, limit, worker) {
  const results = [];
  let i = 0, active = 0, peak = 0;
  return new Promise(resolve => {
    const next = () => {
      if (i >= items.length && active === 0) return resolve({ results, peak });
      while (active < limit && i < items.length) {
        const item = items[i++]; active++; peak = Math.max(peak, active);
        Promise.resolve(worker(item)).then(r => { results.push(r); active--; next(); });
      }
    };
    next();
  });
}
const { peak } = await pool([1,2,3,4,5,6,7,8], 3, x => x * x);
console.log('8 tasks, limit 3 → peak concurrency never exceeded', peak);

4Race conditions live in shared state

JS won't interrupt a function mid-line, but await is a yield point — state can change while you're "away". Two overlapping async operations reading-then-writing the same value corrupt it.

JavaScript · runnable
let balance = 100;
const sleep = ms => new Promise(r => setTimeout(r, ms));

async function withdraw(amt) {
  const current = balance;   // read
  await sleep(10);            // ← yield: the other withdraw runs here
  balance = current - amt;    // write stale value
}
await Promise.all([withdraw(100), withdraw(100)]);
console.log('balance:', balance, '— expected -100, got 0. Lost update!');
console.log('fix → serialize with a lock, or an atomic DB update (Phase 2)');

5Debounce & throttle

Two rate-shaping tools you'll reach for constantly. Debounce: wait until activity stops (search-as-you-type). Throttle: at most once per interval (scroll handlers).

JavaScript · runnable
function debounce(fn, ms) {
  let t;
  return (...a) => { clearTimeout(t); t = setTimeout(() => fn(...a), ms); };
}
let calls = 0;
const search = debounce(q => { calls++; console.log('🔍 API call for:', q); }, 50);

// user types 5 keystrokes quickly...
['r','re','red','redi','redis'].forEach(search);
await new Promise(r => setTimeout(r, 80));
console.log('total API calls:', calls, '(only the last keystroke fired)');

6CPU work & backpressure

  • Don't block the loop. A heavy CPU task (image resize, big JSON, crypto) freezes all requests. Offload to worker threads or a queue (The Streaming Mind).
  • Backpressure — when producers outpace consumers, buffer and slow the source (streams' pause()/drain), or you'll exhaust memory.
  • Cancellation — use AbortController to cancel in-flight work when a request disconnects.
Try it: In the Phase 1 API, take an endpoint that fetches from 3 sources sequentially and switch it to Promise.all; measure the latency drop. Then add a concurrency-limited pool to a bulk endpoint so it never opens more than N DB connections at once.
