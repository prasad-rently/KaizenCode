🛡️ The Bulwark

In a distributed system, failure isn't an edge case — it's the steady state. Networks blip, dependencies die, disks fill. The Bulwark is the set of patterns that keep your service standing when the things around it fall over.

1Assume everything fails

Every network call can hang, error, or return garbage. The senior mindset: for each external call, ask "what happens when this is slow, down, or wrong?" — and have an answer before it happens.

FailurePattern that saves you
Call hangs foreverTimeout
Transient blipRetry with backoff + jitter
Dependency is down hardCircuit breaker
One slow dep starves the restBulkhead isolation
Non-critical feature failsGraceful degradation / fallback

2Timeouts — never wait forever

A call with no timeout can pin a request (and its thread/connection) indefinitely. Always bound it. Promise.race against a timer is the primitive:

JavaScript · runnable
const timeout = (p, ms) => Promise.race([
  p,
  new Promise((_, rej) => setTimeout(() => rej(new Error('timeout')), ms)),
]);
const slow = new Promise(r => setTimeout(() => r('ok'), 500));
try {
  console.log(await timeout(slow, 100));   // dep takes 500ms, we wait 100
} catch (e) {
  console.log('⛔ gave up:', e.message, '— fail fast, free the resource');
}

3Retry with exponential backoff + jitter

Retrying immediately hammers a struggling service. Back off exponentially (1s, 2s, 4s…) and add jitter (randomness) so a thousand clients don't retry in lockstep — the "thundering herd".

JavaScript · runnable
async function retry(fn, tries = 5) {
  for (let i = 0; i < tries; i++) {
    try { return await fn(); }
    catch (e) {
      if (i === tries - 1) throw e;
      const base = 2 ** i * 100;              // 100, 200, 400, 800ms
      const wait = base + Math.random() * base; // + jitter
      console.log(`attempt ${i+1} failed → wait ${wait.toFixed(0)}ms`);
      await new Promise(r => setTimeout(r, wait));
    }
  }
}
let n = 0;
const flaky = () => (++n < 3 ? Promise.reject(new Error('503')) : Promise.resolve('✅ success'));
console.log(await retry(flaky));
Only retry idempotent operations (The Gateway / Streaming Mind). Retrying a non-idempotent POST /charge can bill the customer three times. Use an idempotency key.

4The circuit breaker

If a dependency is clearly down, stop calling it — every doomed call wastes time and resources. A breaker trips OPEN after N failures, rejects instantly for a cooldown, then goes HALF-OPEN to test recovery.

JavaScript · runnable
function breaker(fn, threshold = 3) {
  let fails = 0, state = 'CLOSED';
  return async () => {
    if (state === 'OPEN') return '⚡ rejected instantly (circuit OPEN)';
    try {
      const r = await fn();
      fails = 0; return r;
    } catch (e) {
      if (++fails >= threshold) state = 'OPEN';
      return '❌ failed (' + fails + ' in a row) — state: ' + state;
    }
  };
}
const down = () => Promise.reject(new Error('dead'));
const call = breaker(down);
for (let i = 0; i < 5; i++) console.log('call ' + (i+1) + ': ' + await call());
// after 3 failures it stops trying and fails fast

5Bulkheads & graceful degradation

  • Bulkhead — isolate resources per dependency (separate connection pools/thread limits) so one slow downstream can't consume every connection and sink the whole app. Named after a ship's watertight compartments.
  • Graceful degradation — when a non-critical dependency fails, serve a fallback instead of erroring. Recommendations down? Show popular items. Cache miss + DB slow? Serve slightly stale data.
  • Fail open vs fail closed — decide deliberately: a down auth service should fail closed (deny); a down "related articles" widget should fail open (hide it).
JavaScript · runnable
// Fallback: never let a non-critical failure break the page
async function withFallback(primary, fallback) {
  try { return await primary(); }
  catch { return fallback; }
}
const recos = () => Promise.reject(new Error('reco service down'));
const result = await withFallback(recos, ['Popular item A', 'Popular item B']);
console.log('shown to user:', result, '(degraded, but not broken)');

6Chaos engineering

Don't wait for prod to prove your resilience. Inject failure on purpose — kill instances, add latency, drop packets — in a controlled way, and verify the system copes. If you've never tested a failure, you haven't handled it.

Mobile analogy: this is your app's offline/airplane-mode and flaky-network handling — retries, cached data, "you're offline" states — but applied to every server-to-server call. You already think this way for mobile; apply it to the backend.
Try it: Wrap an outbound call in the Phase 3 app with (1) a 2s timeout, (2) 3 retries with backoff + jitter, and (3) a circuit breaker. Then add a fallback response so the page still renders when the dependency is down. Test it by pointing at a dead URL.
