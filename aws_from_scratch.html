1What is AWS, really?

Strip away the jargon and AWS (Amazon Web Services) is this: Amazon owns millions of computers in giant buildings around the world, and rents them to you by the second. Everything else — all 200+ services — is packaging around that one idea.

Before the cloud, launching an app meant buying physical servers, racking them in a data center, wiring power and networking, and guessing capacity months ahead. Guess low and your app dies on launch day; guess high and you burn money on idle metal. AWS turned all of that into an API call: ask for a computer, get one in 60 seconds, pay only while it runs, hand it back when done.

The mobile analogy you already live: you don't run your own push-notification servers — you use APNs. You don't build your own crash reporting — you drop in a framework. AWS applies that same move to everything: computing, storage, databases, networking. It's "someone else's data center" behind clean APIs, the way APNs is "someone else's push infrastructure".

Three properties make it more than a hosting company:

  • Elasticity — go from 1 server to 100 for launch day, back to 1 on Monday. You can't do that with metal you own.
  • Pay-as-you-go — no upfront cost, no contracts (unless you want discounts). A side project costs pennies; you pay for what you use, metered per second/GB/request.
  • Managed services — beyond raw computers, AWS runs databases, queues, and DNS for you: they patch, back up, and failover, you just use the endpoint. That's the real time-saver.
Mental model for the whole page: every AWS service answers one question. EC2: "where does my code run?" · S3: "where do my files live?" · RDS: "where does my data live?" · IAM: "who's allowed to do what?" · VPC: "how is it all networked?" · Route 53: "how do users find it?" · CloudWatch: "how do I know it's working?"

2The map — regions, availability zones, edge

AWS's physical world has three layers. Understanding them explains half of AWS's vocabulary:

LayerWhat it physically isWhy it exists
Region (e.g. us-east-1, ap-south-1 Mumbai)A cluster of data centers in one geographic area. ~35 worldwide.Put servers near your users (latency) and satisfy data-residency law (e.g. Indian user data staying in India).
Availability Zone (AZ, e.g. ap-south-1a)One or more separate buildings inside a region — own power, cooling, network — linked by fast private fiber.Survive a fire or power cut in one building. Run in 2+ AZs and one building can literally burn down without downtime.
Edge locationSmall cache sites in 400+ cities.Serve static files (CloudFront CDN) and DNS (Route 53) from a few ms away, even where there's no full region.

Almost everything you create lives in one region you choose. Pick the region closest to your users and stay in it while learning — resources in one region are invisible from another console view, which is the #1 "where did my server go?!" beginner panic.

JavaScript · runnable
// Why regions matter: speed of light is a real constraint.
// Round-trip time ≈ distance / speed of light in fiber (~200,000 km/s), × 2
const regions = [
  { name: 'ap-south-1 (Mumbai)',    km: 1200 },  // from Chennai
  { name: 'ap-southeast-1 (Singapore)', km: 2900 },
  { name: 'eu-west-1 (Ireland)',    km: 8600 },
  { name: 'us-east-1 (Virginia)',   km: 13500 },
];
for (const r of regions) {
  const rttMs = (r.km / 200000) * 2 * 1000;
  console.log(r.name.padEnd(28), '~' + rttMs.toFixed(0) + 'ms minimum round trip');
}
console.log('\nNo code optimization beats picking the right region.');

  • The CLI — aws in your terminal. Every button in the console is also a command. Scriptable, repeatable, and what this page's labs use once you're comfortable.
  • SDKs & Infrastructure-as-Code — call AWS from code (@aws-sdk/client-s3 in Node) or declare whole environments in Terraform/CDK (that's The Blueprint). Real teams live here; clicking doesn't scale.

4Money first — free tier & the billing alarm

Do this before creating anything else. AWS bills for what you leave running, and novices get burned by forgotten resources, not by usage.

  • Free tier: new accounts get 12 months of a small server (750 hrs/month of t2.micro/t3.micro EC2 — enough to run one 24/7), 5 GB of S3, 750 hrs of a small RDS database, and 1M Lambda requests/month forever. Everything in this page's planner fits in it.
  • The two rules: ① always know what you have running (one region!), ② delete what you're done with — the teardown checklist is part of the planner.
Non-negotiable first task: in the console → Billing → Budgets, create a budget with an email alert at $5. Five minutes of work; converts every future mistake from a scary invoice into a polite email. The planner makes this Day 1.

Shared responsibility — one more orientation idea: AWS secures the cloud (buildings, hardware, hypervisor); you secure what's in it (your OS patches, your open ports, your IAM keys, your data). When something leaks from an S3 bucket in the news, it's almost always the customer's half.

5IAM — who may do what

What it is: Identity and Access Management — AWS's permission system. Every single API call (console click, CLI command, SDK call) is checked against IAM: is this identity allowed to do this action on this resource?

The four nouns

NounWhat it isMobile analogy
Root accountThe email you signed up with. Can do anything, including delete the account.Your Apple ID — you don't paste it into every tool; you lock it away with MFA and stop using it day-to-day.
UserA person identity with a password and/or access keys.A team member's App Store Connect login.
RoleAn identity with no password that a service or user temporarily assumes. How an EC2 server gets S3 access without any stored secret.A provisioning profile — the app carries a capability, no human credential embedded.
PolicyA JSON document listing allowed/denied actions. Attach to users/roles.The entitlements file.

Anatomy of a policy

JSON · copy
{
  "Version": "2012-10-17",
  "Statement": [{
    "Effect":   "Allow",
    "Action":   ["s3:GetObject", "s3:PutObject"],   // which API calls
    "Resource": "arn:aws:s3:::my-app-uploads/*"       // on which things
  }]
}

How evaluation works (this ordering is the whole trick): start from deny everything → an Allow anywhere grants it → but an explicit Deny anywhere always wins, even over an Allow. Run it:

JavaScript · runnable
// A miniature IAM evaluator — the real one works exactly like this
const policies = [
  { effect: 'Allow', action: 's3:GetObject', resource: 'my-app-uploads/*' },
  { effect: 'Allow', action: 's3:PutObject', resource: 'my-app-uploads/*' },
  { effect: 'Deny',  action: 's3:*',         resource: 'my-app-uploads/secrets/*' },
];
const match = (pat, val) => new RegExp('^' + pat.replace(/[.+?^${}()|[\]\\]/g,'\\$&').replace(/\*/g,'.*') + '$').test(val);

function evaluate(action, resource) {
  const hits = policies.filter(p => match(p.action, action) && match(p.resource, resource));
  if (hits.some(p => p.effect === 'Deny'))  return '⛔ DENY (explicit deny wins)';
  if (hits.some(p => p.effect === 'Allow')) return '✅ ALLOW';
  return '⛔ DENY (default — nothing allowed it)';
}
console.log('read a photo   →', evaluate('s3:GetObject', 'my-app-uploads/photos/1.jpg'));
console.log('read a secret  →', evaluate('s3:GetObject', 'my-app-uploads/secrets/key.pem'));
console.log('delete a photo →', evaluate('s3:DeleteObject', 'my-app-uploads/photos/1.jpg'));
Least privilege: grant the narrowest permission that works — s3:GetObject on one bucket, not s3:* on everything. It feels pedantic until leaked keys can only read one bucket instead of deleting your company.
🧪 TRY IT — 15 min, free
  1. Sign in as root → enable MFA on the root account (IAM dashboard tells you to).
  2. IAM → Users → Create user admin-you, console access, attach the AdministratorAccess policy. Sign out of root, in as this user. Root now retires to a drawer.
  3. Create an access key for CLI use, run aws configure, then aws sts get-caller-identity — seeing your user's ARN means the whole chain works.

6EC2 — renting computers

What it is: Elastic Compute Cloud — the original AWS service. An EC2 instance is a virtual computer: you pick its size, choose an operating system, press launch, and ~40 seconds later you can SSH into a Linux box that lives in an AWS building.

How it actually works

AWS doesn't give you a whole physical machine. A physical server runs a hypervisor (AWS's is called Nitro) that slices it into isolated virtual machines. Your "computer" is one slice: the hypervisor gives it dedicated vCPUs and memory and enforces that neighbors can't see each other. Virtualization is why your instance boots in seconds (it's just a process starting, not hardware powering on) and why you can resize it (your slice is re-carved elsewhere).

The pieces that make up one instance:

PieceWhat it is
AMI (Amazon Machine Image)The disk template it boots from — OS + pre-installed software. "Amazon Linux 2023" or "Ubuntu 24.04" to start. You can snapshot your configured box into your own AMI and stamp out clones.
Instance typeThe size. The name encodes it: t3.micro = family t (burstable, cheap) · generation 3 · size micro (2 vCPU, 1 GB RAM). Families: t/m general, c compute-heavy, r memory-heavy, g/p GPU.
EBS volumeIts hard drive — network-attached, so it survives the instance and can detach/reattach like an external SSD.
Key pairSSH keys. AWS keeps the public key, you download the private .pem — the only way in. Lose it and you're locked out.
Security groupA firewall around the instance: "allow TCP 22 from my IP, TCP 443 from anywhere". Stateful — replies to allowed traffic flow back automatically. Default inbound: block everything.

Lifecycle & pricing

Stop an instance and compute billing stops (the EBS disk still costs ~$0.08/GB-month; your files survive). Terminate deletes it, disk and all. Pricing has three flavors — on-demand (default), reserved/savings plans (commit 1–3 yrs, ~40% off), and spot (spare capacity, ~70% off, reclaimable with 2-min warning — fine for batch work, wrong for your only web server). The Wallet goes deeper.

  2. Connect: chmod 400 lab-1.pem && ssh -i lab-1.pem ec2-user@<public-ip>. That prompt is a computer in an AWS building.

7S3 — infinite object storage

What it is: Simple Storage Service — a place to put files (S3 calls them objects): images, videos, backups, logs, entire static websites. You don't provision size; it holds one file or a trillion, billed ~$0.023/GB-month for what you actually store.

How it actually works

  • Not a hard drive. S3 is a key→value store over HTTP. A bucket (name globally unique across all AWS customers) holds objects, each identified by a key like users/42/avatar.png. There are no real folders — the console just draws slashes as folders. You PUT, GET, DELETE whole objects; you can't edit byte 5 of a file in place.
  • The famous 11 nines: 99.999999999% durability. Every object is automatically copied across at least 3 availability zones — three separate buildings must fail simultaneously to lose your file. This is why "put it in S3" is the answer to "where do backups go?".
  • Storage classes trade retrieval speed for price: Standard (hot) → Standard-IA (rarely read, ~45% cheaper) → Glacier (archives, ~90% cheaper, minutes–hours to retrieve). Lifecycle rules move objects automatically as they age.
  • Access: buckets are private by default. Grant access via IAM policies, or hand out a presigned URL — a time-limited signed link that lets anyone holding it download/upload exactly one object. That's how apps do user uploads without proxying bytes through their server.
JavaScript · runnable
// S3 in 25 lines — a bucket is literally a Map with prefix queries
class Bucket {
  #objects = new Map();
  put(key, body)  { this.#objects.set(key, { body, size: body.length }); }
  get(key)        { return this.#objects.get(key)?.body ?? '404 NoSuchKey'; }
  list(prefix='') { return [...this.#objects.keys()].filter(k => k.startsWith(prefix)); }
}
const b = new Bucket();
b.put('users/42/avatar.png', '…png bytes…');
b.put('users/42/cover.jpg',  '…jpg bytes…');
b.put('logs/2026-07-12.txt',  'GET /health 200');

console.log('GET users/42/avatar.png →', b.get('users/42/avatar.png'));
console.log('LIST users/42/ →', b.list('users/42/'));
console.log('GET missing.txt →', b.get('missing.txt'));
console.log('\n"Folders" are a lie — users/42/ is just a key prefix.');
🧪 TRY IT — host a website for pennies, 20 min
  1. S3 → Create bucket yourname-first-site (pick your region, leave "block public access" ON for now).
  2. Upload any file from the console, then fetch it from your terminal with the CLI:
    Shell · copy
    echo '<h1>deployed from my laptop</h1>' > index.html
aws s3 cp index.html s3://yourname-first-site/
aws s3 ls s3://yourname-first-site/          # there it is
aws s3 presign s3://yourname-first-site/index.html --expires-in 300
  3. Open the presigned URL it prints — you're reading a private object through a 5-minute signed link. Wait 5 minutes, refresh: AccessDenied. That's presigning.
  4. Bonus: enable "Static website hosting" on the bucket (and disable block-public-access + add a public-read bucket policy) and your index.html is a live website. This exact setup — S3 + CloudFront in front — serves half the internet's landing pages, including this site via GitHub Pages' equivalent.

8VPC — your private network

What it is: Virtual Private Cloud — your own private network inside AWS, where EC2 instances and RDS databases actually live. Every account gets a default VPC (which is why the EC2 lab "just worked"), but understanding it is what separates copy-pasters from engineers.

Networking from absolute scratch

  • Every device on a network has an IP address (10.0.1.17). A subnet is a contiguous block of them.
  • Blocks are written in CIDR notation: 10.0.0.0/16 means "the first 16 bits are fixed" → 65,536 addresses (10.0.x.x). A /24 fixes 24 bits → 256 addresses. Smaller suffix = bigger block.
  • A VPC is one big block (say 10.0.0.0/16) that you slice into subnets, each pinned to one availability zone.
JavaScript · runnable
// CIDR decoder — the mystery notation, demystified
function cidr(block) {
  const [ip, bits] = block.split('/');
  const hostBits = 32 - Number(bits);
  const total = 2 ** hostBits;
  return { block, usable: total - 5, note: 'AWS reserves 5 per subnet' };
}
['10.0.0.0/16', '10.0.1.0/24', '10.0.2.0/28'].forEach(b =>
  console.log(b.padEnd(14), '→', cidr(b).usable.toLocaleString().padStart(6), 'usable IPs'));
console.log('\nTypical layout: VPC /16, one public + one private /24 per AZ.');

Public vs private — the pattern behind every real architecture

PieceJob
Internet Gateway (IGW)The VPC's door to the internet.
Route tableEach subnet's signpost. A public subnet's table routes 0.0.0.0/0 (everywhere) → IGW. A private subnet has no such route — nothing outside can reach it, ever.
NAT GatewayLets private machines dial out (for updates, APIs) while remaining unreachable from outside. (~$32/mo — the classic surprise bill; skip it while learning.)
Security group vs NACLSG = stateful firewall on the instance (your main tool). NACL = stateless filter on the subnet (leave at default until you need it).
The golden pattern: load balancer in public subnets; app servers and the database in private subnets. Your database should not have a public IP address. Most real-world breaches trace to someone violating this one sentence.

9RDS — managed databases

What it is: Relational Database Service — AWS runs PostgreSQL/MySQL for you. You could install Postgres on an EC2 box yourself (it's just Linux), so RDS's entire value is in what you stop doing:

Self-hosted on EC2 (you do it)RDS (a checkbox)
Install, patch, upgrade PostgresAutomatic during a maintenance window you pick
Cron nightly pg_dump, test restores, store offsiteContinuous backups; point-in-time restore to any second in the last 35 days
Build failover: replica, health checks, promotion scripts, 3am pagerMulti-AZ: a hot standby in another building; automatic failover in ~60s, same DNS name
Scale reads yourself with replication configRead replicas — a checkbox, up to 15

This is "managed services" in its purest form, and the general cloud trade: pay ~30% more per hour, delete an entire category of ops work. For a small team, it's the difference between having a DBA and not needing one. (What Core Data is to writing your own SQLite threading layer, RDS is to running your own database server.)

🧪 TRY IT — 30 min, free tier
  1. RDS → Create database → PostgreSQL → Free tier template (db.t3.micro, single-AZ, 20 GB). Set a master password. Public access: No.
  2. Note it has no public IP — by design. Reach it from your EC2 lab box instead: add an inbound rule on the DB's security group allowing TCP 5432 from the EC2 instance's security group (SGs can reference each other — this is the idiomatic pattern).
  3. From the EC2 box: sudo dnf install -y postgresql16 && psql -h <rds-endpoint> -U postgres → CREATE TABLE hello(id int); You're using a database that backs itself up and can survive a building failure.
  4. Delete it when done (planner). Skip the final snapshot for a lab.

10Lambda — serverless functions

What it is: upload a function; AWS runs it when something happens — an HTTP request, a file landing in S3, a queue message, a timer. No server to create, patch, or pay for while idle. Billed per invocation and per millisecond of runtime; 1M requests/month free forever.

How it actually works

There is a server, of course — you just never see it. On invocation AWS spins up a Firecracker microVM (a tiny VM that boots in ~125ms), loads your code, runs the handler, then keeps the warm VM around a few minutes for reuse. First hit pays the cold start; subsequent hits are warm and fast. Traffic spike? AWS just starts more microVMs in parallel — scaling is automatic and effectively unbounded.

JavaScript · runnable
// Feel the cold-start / warm-start difference
let warmVM = null;
async function invoke(n) {
  const t0 = performance.now();
  if (!warmVM) {                       // cold: boot microVM + load code
    await new Promise(r => setTimeout(r, 180));
    warmVM = { loadedAt: n };
  }
  await new Promise(r => setTimeout(r, 8));  // your actual handler work
  return (performance.now() - t0).toFixed(0);
}
for (let i = 1; i <= 4; i++)
  console.log('request', i, '→', await invoke(i) + 'ms', i === 1 ? '  ← cold start' : '  (warm)');
console.log('\nRule: great for spiky/event work; a 24/7 hot API path may prefer EC2.');

A real handler is just this — and with a Function URL enabled, it's a public HTTPS endpoint with zero other infrastructure:

JavaScript · Lambda handler · copy
export const handler = async (event) => {
  const name = event.queryStringParameters?.name ?? 'world';
  return { statusCode: 200, body: JSON.stringify({ hello: name }) };
};
🧪 TRY IT — an API with no server, 15 min, free
  1. Lambda → Create function hello-api, Node.js runtime. Paste the handler above into the inline editor, Deploy.
  2. Configuration → Function URL → create one (auth type NONE for the lab).
  3. curl "<function-url>?name=you" — an HTTPS API that costs $0 while idle. Invoke it 100× in a loop and notice request 1 is slower: your cold start, live.

11Route 53, CloudFront & CloudWatch — finding it, speeding it, watching it

Route 53 — DNS

DNS is the internet's phone book: it turns api.yourapp.com into an IP address. Route 53 hosts your domain's records: A (name → IP), CNAME (name → another name), and AWS's special Alias (name → an AWS resource like a load balancer or CloudFront). Beyond the basics it does health-checked failover (stop routing to a dead region) and latency-based routing (send each user to the nearest one).

CloudFront — the CDN

A layer of 400+ edge caches in front of your site. A user in Chennai hits the Chennai edge, which serves cached copies of your S3 files or API responses instead of crossing the ocean — the ~200ms physics tax from section 2 drops to ~10ms. S3 + CloudFront is the canonical static-site stack; it also terminates HTTPS with a free certificate.

CloudWatch — eyes on everything

Every AWS service emits metrics (CPU, request counts, errors) into CloudWatch; your apps ship logs to it; alarms watch any metric and act — email you, or trigger autoscaling. The starter kit: a CPU alarm on your EC2 box, and the billing alarm from section 4 (which is itself a CloudWatch alarm). The Watchtower turns this into a discipline.

15Teardown — the don't-get-billed checklist

The bill comes from what you forget, not what you use. After any lab session, sweep this list (in your one region):
  • EC2 → Instances: terminate labs (or stop, accepting ~$0.80/mo per 10 GB disk). Check Elastic IPs — an unattached one costs money; release it.
  • RDS → delete lab databases (skip final snapshot for throwaways).
  • EBS → Volumes: delete anything "available" (= orphaned disk).
  • NAT Gateways / Load balancers → the two classic ~$20–30/mo stowaways; delete if you experimented.
  • S3 → pennies; fine to keep, or empty + delete the bucket.
  • Lambda / IAM / security groups → free to keep.
  • Finally: Cost Explorer → confirm the daily spend graph is flat at ~zero, and trust your $5 alarm as the backstop.
