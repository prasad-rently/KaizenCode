🔒 AES-GCM — Authenticated Symmetric Encryption

The workhorse that actually encrypts your data. One shared key locks and unlocks the box, it's blisteringly fast, and a built-in tamper seal means a modified message is rejected, not silently mis-decrypted.

Part of the Cryptography topic. ECDH agrees a shared key; HMAC proves integrity. AES-GCM is where the bulk data actually gets encrypted — and it folds integrity in for free. It's the "symmetric" half of the hybrid encryption every HTTPS connection uses.

1Symmetric vs asymmetric

Symmetric encryption uses one key for both locking and unlocking — like a physical lockbox where both people hold a copy of the same key. It's fast and handles unlimited data, but both sides must already share that key.

Asymmetric (RSA, ECDH) uses a public/private key pair and solves the "how do we share a key without meeting" problem — but it's slow and can only handle tiny amounts of data.

So real systems are hybrid: use asymmetric crypto once to agree a key, then symmetric AES-GCM for all the actual data. That's exactly the ECIES flow from the ECDH module.

2What AES-GCM is made of

AES is a block cipher — it scrambles data in 16-byte blocks using a key of 128, 192, or 256 bits. On its own it just transforms blocks; you need a mode to encrypt real-world messages. GCM (Galois/Counter Mode) is that mode, and it does two jobs at once:

  • Counter mode turns AES into a stream cipher, so you can encrypt data of any length efficiently and in parallel.
  • An authentication tag (16 bytes appended to the ciphertext) that acts as a tamper seal — this is what makes GCM an AEAD (Authenticated Encryption with Associated Data) cipher.

Encrypting takes three inputs and produces two outputs:

   KEY  (secret, shared)          ─┐
   IV / NONCE  (fresh per message) ├──▶  AES-GCM  ──▶  CIPHERTEXT + AUTH TAG
   PLAINTEXT                      ─┘

The IV (initialization vector, a.k.a. nonce — "number used once") is a fresh 12-byte value per message. It's not secret — you send it alongside the ciphertext — but it must be unique for every message under a given key. Its job: make identical plaintexts encrypt to different ciphertexts, so an eavesdropper can't even tell you sent the same thing twice.

3See it: encrypt, decrypt, and the IV's job

Below we encrypt the same message twice with two fresh IVs — the ciphertexts come out completely different — then decrypt one back. Run it:

JavaScript · runnable (WebCrypto)
// AES-256-GCM: generate a key, encrypt, decrypt
const enc = new TextEncoder(), dec = new TextDecoder();
const hex = u => [...new Uint8Array(u)]
  .map(b => b.toString(16).padStart(2, '0')).join('');

const key = await crypto.subtle.generateKey(
  { name: 'AES-GCM', length: 256 }, true, ['encrypt', 'decrypt']);

const msg = 'Meet at the docks at midnight';

// Encrypt the SAME message twice, each with its own fresh 96-bit IV
const iv1 = crypto.getRandomValues(new Uint8Array(12));
const iv2 = crypto.getRandomValues(new Uint8Array(12));
const c1 = await crypto.subtle.encrypt({ name: 'AES-GCM', iv: iv1 }, key, enc.encode(msg));
const c2 = await crypto.subtle.encrypt({ name: 'AES-GCM', iv: iv2 }, key, enc.encode(msg));
console.log('ciphertext #1:', hex(c1).slice(0, 40) + '…');
console.log('ciphertext #2:', hex(c2).slice(0, 40) + '…');
console.log('same plaintext → different ciphertext?', hex(c1) !== hex(c2));
console.log('ciphertext bytes:', new Uint8Array(c1).length, '(plaintext + 16-byte tag)');

// Decrypt #1 — needs the same key AND the matching IV
const back = await crypto.subtle.decrypt({ name: 'AES-GCM', iv: iv1 }, key, c1);
console.log('decrypted #1 :', dec.decode(back));

4The tamper seal (AEAD) — and "associated data"

Because GCM authenticates the ciphertext, flipping even a single bit makes decryption fail loudly rather than return corrupted plaintext. You can also bind extra associated data (AAD) to the ciphertext — data that travels in the clear but is authenticated, like a message header or a user ID. Change the AAD and decryption also fails. Run it:

JavaScript · runnable (WebCrypto)
// AES-GCM rejects any tampering — with the ciphertext OR the associated data
const enc = new TextEncoder(), dec = new TextDecoder();
const key = await crypto.subtle.generateKey(
  { name: 'AES-GCM', length: 256 }, false, ['encrypt', 'decrypt']);
const iv = crypto.getRandomValues(new Uint8Array(12));
const aad = enc.encode('user-id:42');        // authenticated, NOT encrypted

const ct = await crypto.subtle.encrypt(
  { name: 'AES-GCM', iv, additionalData: aad }, key, enc.encode('balance=100'));

// 1) Flip one byte of the ciphertext (use .slice() to copy, not view!)
const tampered = new Uint8Array(ct).slice();
tampered[0] ^= 0x01;
try {
  await crypto.subtle.decrypt({ name: 'AES-GCM', iv, additionalData: aad }, key, tampered);
  console.log('tampered ciphertext → SUCCEEDED (should never happen)');
} catch (e) {
  console.log('tampered ciphertext → rejected:', e.name);
}

// 2) Correct ciphertext but WRONG associated data
try {
  await crypto.subtle.decrypt(
    { name: 'AES-GCM', iv, additionalData: enc.encode('user-id:99') }, key, ct);
  console.log('wrong AAD → SUCCEEDED (should never happen)');
} catch (e) {
  console.log('wrong AAD → rejected:', e.name);
}

// 3) Intact ciphertext + correct AAD → decrypts cleanly
const ok = await crypto.subtle.decrypt({ name: 'AES-GCM', iv, additionalData: aad }, key, ct);
console.log('intact + correct AAD →', dec.decode(ok));

5The one rule you must never break

Never reuse an IV/nonce with the same key. GCM's security collapses if you encrypt two different messages with the same (key, IV) pair: an attacker can XOR the two ciphertexts to strip the keystream and recover relationships between the plaintexts — and can forge the authentication tag for that key. This is the single most common way real systems using GCM get broken.

Safe practice: generate a fresh random 12-byte IV with a CSPRNG (crypto.getRandomValues) for every message, or use a strict counter you're certain never repeats. The IV isn't secret — just unique. (If you can't guarantee uniqueness, use a nonce-misuse-resistant mode like AES-GCM-SIV instead.)

7Modes in action — why ECB leaks patterns

A raw block cipher only encrypts one 16-byte block. To encrypt a whole message you need a mode of operation that says how the blocks relate. The naïve choice, ECB (Electronic Codebook), encrypts every block independently — so identical plaintext blocks become identical ciphertext blocks, and any repetition in your data shows straight through. The classic proof: encrypt an image in ECB and you can still see the picture.

CBC fixes the leak by XORing each block with the previous ciphertext block before encrypting, seeded by a random IV (initialization vector) that must be unpredictable and unique per message. But CBC gives confidentiality only — no built-in tamper detection — and, handled carelessly, enables padding-oracle attacks (a server that reveals whether padding was valid lets an attacker decrypt byte by byte). That's why authenticated modes win: never ship CBC without encrypt-then-MAC, or just use an AEAD like GCM.

8Stream ciphers & ChaCha20

In one sentence: a stream cipher produces a river of pseudo-random bytes — a keystream — and XORs it with your data. Fast, and safe even without special hardware.

Where a block cipher locks data into fixed-size safes, a stream cipher works like a usable one-time pad: generate an endless keystream from the key, XOR it over the plaintext to encrypt, and XOR the same keystream over the ciphertext to decrypt. Because XOR is its own inverse, the two directions are the identical operation.

Why ChaCha20 is special

  • Fast in software alone. AES leans on CPU instructions (AES-NI); without them — many phones, ARM, embedded chips — AES slows down, but ChaCha20 stays fast. That's why Google adopted ChaCha20-Poly1305 for TLS on devices lacking AES hardware.
  • Constant-time by design. Its add-rotate-XOR (ARX) core never branches on secret data, sidestepping the timing side-channels that have bitten some AES software implementations.
  • Simple and auditable. The whole core is additions, rotations, and XORs — easy to implement correctly.

In practice it's always used as ChaCha20-Poly1305, an AEAD that adds a Poly1305 authentication tag — exactly the role GCM plays for AES. Honest tradeoff: with AES hardware, AES-GCM is often faster; ChaCha20 wins when there's no AES acceleration, or when software side-channel safety matters.

9Retired ciphers — and why

A cipher isn't retired because the idea was foolish — it's retired because of key size, block size, or flaws found over years of public analysis. That's precisely why we trust openly analysed standards over anything homemade. These three are read-only history; Web Crypto omits them by design:

10Where you'll meet AES-GCM

  • TLS 1.3 / HTTPS — AES_256_GCM is a default cipher suite; it encrypts the actual bytes of every request after the ECDHE handshake.
  • Disk & file encryption — FileVault, BitLocker, and mobile-device encryption.
  • The payload half of ECIES — the ECDH module derives an AES-GCM key with HKDF, then encrypts with exactly this.
  • At-rest encryption — S3 SSE, database and secrets-manager encryption, encrypted backups.
  • VPNs & messaging — WireGuard and Signal use AEAD ciphers (ChaCha20-Poly1305, GCM's cousin).
