📐 The Blueprint

Phase 5 created AWS resources by clicking. That doesn't scale, can't be reviewed, and can't be reproduced. Infrastructure as Code makes your servers a file in git.

1Declarative, not imperative

You don't write steps ("create a server, then attach a disk"). You declare the desired end state, and the tool figures out the diff between what exists and what you want.

Click-ops (Phase 5)Infrastructure as Code
Manual, error-proneReproducible from one file
No history of changesEvery change is a git diff + PR review
"Works in prod, not staging"Same code → identical environments
Knowledge in someone's headKnowledge in the repo
Mobile analogy: click-ops is laying out a screen by dragging in Interface Builder with no record of why. IaC is SwiftUI — the UI is the declarative code, diffable and reviewable.

2Terraform — the same EC2 + S3 from Phase 5, as code

Read this top-to-bottom: it declares exactly the resources you clicked together before — now version-controlled.

HCL · main.tf · copy
provider "aws" {
  region = "ap-south-1"
}

# An S3 bucket for uploads (Phase 5, §S3)
resource "aws_s3_bucket" "uploads" {
  bucket = "kaizencode-uploads"
  tags   = { Project = "KaizenCode" }
}

# A web server (Phase 5, §EC2)
resource "aws_instance" "api" {
  ami           = "ami-0abcd1234"
  instance_type = "t3.micro"
  tags          = { Name = "kaizencode-api" }
}

# Output the public IP after apply
output "api_ip" {
  value = aws_instance.api.public_ip
}

3The core workflow: init → plan → apply

Shell · copy
terraform init      # download the aws provider
terraform plan      # DRY RUN — shows exactly what will change
terraform apply     # make reality match the file
terraform destroy   # tear it all down (great for learning, no surprise bills)
Always read the plan. It prints + create, ~ update, - destroy per resource. A stray - destroy on your database is how outages happen. Plan is your seatbelt.

4State & drift

Terraform records what it created in a state file. "Drift" is when reality diverges from that state — someone changed a setting in the console by hand. Run this to see how a plan is just a three-way diff.

JavaScript · runnable
// what terraform plan does: compare desired vs real
const desired = { instance_type: 't3.micro', count: 2, public: false };
const real    = { instance_type: 't3.small', count: 2, public: true }; // hand-edited!

for (const key of Object.keys(desired)) {
  if (desired[key] !== real[key])
    console.log(`~ ${key}: ${real[key]} → ${desired[key]}  (drift)`);
  else
    console.log(`  ${key}: ${real[key]}  (in sync)`);
}
console.log('\napply would revert the console changes back to code');

5Modules, GitOps & the rules

  • Modules — package a VPC or service into a reusable block; instantiate per environment with different variables.
  • Remote state — store state in S3 + DynamoDB lock (never local, never in git — it can hold secrets).
  • GitOps — infra changes flow through PRs; CI runs plan on the PR, apply on merge. Phase 4's GitHub Actions, pointed at infrastructure.
  • Never edit prod by hand — it causes drift. Change the code, review, apply.
Try it: Install Terraform, copy the main.tf above (use a real AMI for your region), and run init → plan. Read the plan output carefully, then apply, then destroy so you never get a bill. You just managed cloud infra from a file.

Alternatives worth knowing: Pulumi (IaC in real TypeScript/Python), AWS CDK (CloudFormation from code), Ansible (configuration management). Terraform is the most portable starting point.
