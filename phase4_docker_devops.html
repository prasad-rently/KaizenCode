Docker & DevOps

Containerise your app, automate your pipeline, ship with confidence.

1What is Docker?

The classic problem: "It works on my machine." Docker solves this by packaging your app and all its dependencies into a container — a portable, self-contained unit that runs identically on any machine, any OS, any cloud.

Mobile analogy: A Docker image is like an .ipa or .apk — it bundles the app and its runtime together. Docker runs these bundles on any server, the same way iOS runs .ipa on any iPhone.
Virtual Machine
Emulates full hardware. Has its own OS kernel. Heavy (GBs). Slow to start (minutes). Full isolation.
Container (Docker)
Shares host OS kernel. Just app + deps. Lightweight (MBs). Starts in seconds. Process-level isolation.

2Dockerfile — building an image

A Dockerfile is a recipe that defines your container image, layer by layer. Each instruction creates a cached layer — unchanged layers are reused on rebuild.

# Start from the official Node.js 20 base image
FROM node:20-alpine

# Set working directory inside the container
WORKDIR /app

# Copy package files FIRST (separate layer = better caching)
COPY package*.json ./
RUN npm ci --only=production

# Copy source code (only re-runs when source changes)
COPY . .

# Document which port the app listens on
EXPOSE 3000

# Default command when the container starts
CMD ["node", "server.js"]
# Build the image
docker build -t my-api:1.0 .

# Run the container (map host port 3000 to container port 3000)
docker run -p 3000:3000 my-api:1.0

# Run detached (background), with env variable
docker run -d -p 3000:3000 -e NODE_ENV=production my-api:1.0

3Essential Docker commands

docker ps                 # list running containers
docker ps -a              # all containers (including stopped)
docker images             # list local images
docker logs <id>          # view stdout/stderr (like Xcode console)
docker exec -it <id> sh  # open a shell inside running container
docker stop <id>          # gracefully stop
docker rm <id>            # remove stopped container
docker rmi <image>        # remove image
docker system prune       # clean up all unused resources

4Docker Compose — multi-container apps

Real apps need multiple services running together: API server, database, cache, background worker. Docker Compose defines all of them in one YAML file and starts them with a single command.

# docker-compose.yml
version: '3.8'
services:
  api:
    build: .
    ports: ["3000:3000"]
    environment:
      - MONGODB_URI=mongodb://mongo:27017/myapp
      - REDIS_URL=redis://redis:6379
    depends_on: [mongo, redis]
    volumes:
      - .:/app             # hot reload in dev
      - /app/node_modules  # keep container's node_modules

  mongo:
    image: mongo:7
    volumes: [mongo_data:/data/db]
    ports: ["27017:27017"]

  redis:
    image: redis:7-alpine
    ports: ["6379:6379"]

volumes:
  mongo_data:
docker compose up -d        # start all services in background
docker compose down         # stop and remove containers
docker compose logs -f api  # follow logs from api service
docker compose exec api sh  # shell into running api container
Mobile analogy: docker-compose.yml is like a Package.swift or Podfile for your entire runtime environment — one file declares all services, one command boots them all.

5CI/CD — automated pipelines

CI/CD means every push to your repo automatically: runs tests → builds → deploys. No manual steps, no "deploy Fridays", no human error.

CI — Continuous Integration
Run linting + tests on every push. Catch bugs before they merge to main. Fail fast.
CD — Continuous Deployment
If tests pass on main, auto-deploy to staging or production. Removes the human deploy step.
# .github/workflows/ci.yml
name: CI Pipeline
on:
  push:
    branches: [main, develop]
  pull_request:
    branches: [main]

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Setup Node 20
        uses: actions/setup-node@v4
        with: { node-version: '20' }

      - name: Install deps
        run: npm ci

      - name: Lint
        run: npm run lint

      - name: Test
        run: npm test -- --coverage

      - name: Build Docker image
        run: docker build -t my-api:latest .

      # On success → deploy to Render / AWS / etc.

6Nginx — the front door

Nginx (engine-x) sits in front of your Node.js app handling SSL termination, load balancing, and static file serving. Your app never deals with TLS certificates directly.

# /etc/nginx/sites-available/myapp
server {
    listen 80;
    server_name api.myapp.com;
    return 301 https://$host$request_uri;  # force HTTPS
}

server {
    listen 443 ssl;
    server_name api.myapp.com;
    ssl_certificate     /etc/letsencrypt/live/myapp.com/fullchain.pem;
    ssl_certificate_key /etc/letsencrypt/live/myapp.com/privkey.pem;

    location / {
        proxy_pass         http://localhost:3000;  # forward to Node
        proxy_set_header   Host $host;
        proxy_set_header   X-Real-IP $remote_addr;
        proxy_http_version 1.1;
        proxy_set_header   Upgrade $http_upgrade;
        proxy_set_header   Connection 'upgrade';  # for WebSockets
    }
}
Mobile analogy: Nginx is like an API Gateway — it terminates SSL and forwards clean HTTP to your app internally. Your Node.js process only ever sees plain HTTP on localhost.
