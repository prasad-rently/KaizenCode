🚪 The Gateway

Phase 1 taught you REST. Real systems also speak GraphQL and gRPC, sit behind gateways, and must paginate, rate-limit, and version gracefully. This is the front door done right.

1REST vs GraphQL vs gRPC

RESTGraphQLgRPC
ShapeMany URLsOne endpoint, you ask for fieldsBinary RPC over HTTP/2
Over/under-fetchCommonClient picks exact fieldsSchema-defined
Best forPublic, cacheable APIsRich mobile/web clientsService-to-service, low latency
Mobile analogy: with REST you often hit /user then /user/posts then /posts/comments — three round trips. GraphQL lets the app request the whole tree in one query, asking only for the fields the screen renders.

2How a resolver actually works

GraphQL isn't magic — a resolver is just a function per field. Run a tiny GraphQL-style engine: ask for specific fields, get back exactly those.

JavaScript · runnable
// Resolvers: one function per field
const db = { 42: { id: 42, name: 'Gokul', email: 'g@x.io', secret: 'hidden' } };
const resolvers = {
  user: ({ id }) => db[id],
  name: (u) => u.name,
  email: (u) => u.email,
};
// the "query": give me name + email for user 42 (not secret)
const user = resolvers.user({ id: 42 });
const result = {
  name: resolvers.name(user),
  email: resolvers.email(user),
};
console.log(result); // exactly the fields asked for

3The N+1 problem (GraphQL's classic trap)

Naively resolving a list + a nested field fires 1 query for the list, then N more — one per item. At scale this melts your database. Run it to see 1+N vs the batched fix.

JavaScript · runnable
// Simulated DB hits
let naive = 0, batched = 0;
const postIds = [1, 2, 3, 4, 5];

// ❌ N+1: 1 query for posts + 1 per author
naive++;                              // fetch posts
postIds.forEach(() => naive++);      // fetch each author

// ✅ batched (DataLoader): 1 for posts + 1 for ALL authors
batched++;                            // fetch posts
batched++;                            // fetch authors WHERE id IN (...)

console.log('naive queries:  ', naive,   '(1 + N)');
console.log('batched queries:', batched, '(constant)');
console.log('fix → DataLoader: collect ids, one IN-query per tick');

4Cursor pagination (do this, not OFFSET)

Phase 2 used LIMIT/OFFSET. It breaks on live data: insert a row and page 2 shifts, showing duplicates. Cursor pagination ("everything after id X") is stable and fast.

JavaScript · runnable
// Keyset / cursor pagination
const rows = [{id:10},{id:9},{id:8},{id:7},{id:6},{id:5}];
function page(afterId, size) {
  const start = afterId == null
    ? 0
    : rows.findIndex(r => r.id === afterId) + 1;
  const slice = rows.slice(start, start + size);
  return { items: slice, nextCursor: slice.length ? slice[slice.length-1].id : null };
}
const p1 = page(null, 2);
console.log('page 1:', p1);
console.log('page 2:', page(p1.nextCursor, 2)); // stable even if rows inserted

5Rate limiting (the token bucket)

An API gateway's job: protect the backend. The token bucket allows bursts up to a cap, then throttles to a steady refill rate. Run it — the 4th rapid request gets blocked.

JavaScript · runnable
// Token bucket: capacity 3
function makeBucket(capacity) {
  let tokens = capacity;
  return () => {
    if (tokens > 0) { tokens--; return '200 OK   (tokens left: ' + tokens + ')'; }
    return '429 Too Many Requests';
  };
}
const allow = makeBucket(3);
for (let i = 1; i <= 5; i++)
  console.log('req ' + i + ': ' + allow());

6API gateways & versioning

  • Gateway = one front door (Kong, AWS API Gateway, Nginx) handling auth, rate limiting, routing, and TLS — so each service doesn't reimplement them.
  • Versioning — never break existing clients. Add /api/v2 alongside v1; deprecate with a sunset header and a timeline.
  • Idempotency keys — let clients safely retry POST (payments!) without double-charging.
