Distributed Systems & Kafka

How large-scale systems are really built. The CTO's mental model for architecture.

1Why distributed systems?

A single server hits limits — CPU, RAM, disk, network bandwidth. When you exceed those limits, you distribute work across many servers. But distribution introduces new problems: partial failures, network latency, and data consistency challenges.

Vertical scaling ↑
Get a bigger server. Simple — no code changes. But has a hard ceiling, causes downtime to upgrade, and is expensive at the top end.
Horizontal scaling →
Add more servers. No theoretical ceiling — run 1,000 instances. But requires load balancing, shared state management, and coordination.
Mobile analogy: Vertical = upgrading from an iPhone 14 to 15 Pro Max. Horizontal = giving the whole team phones and splitting the calls. The second scales to millions — but you need coordination.

2Message queues — the decoupling pattern

Instead of Service A calling Service B directly (tight coupling — both must be running), A drops a message into a queue and B picks it up when ready (loose coupling — fully independent).

Without a queue (tight)
OrderService → calls EmailService directly. If EmailService is slow or crashes, OrderService also fails. They're coupled at runtime.
With a queue (loose)
OrderService drops "order.placed" event. EmailService picks it up when ready. OrderService never blocks. Neither knows about the other.
Mobile analogy: APNs push notifications. Your backend fires a notification into APNs and moves on immediately. APNs delivers it when the device is online. That IS the message queue pattern — exactly.

3Apache Kafka

Kafka is a distributed message streaming platform. Unlike simple queues (message deleted once consumed), Kafka retains messages for days and lets multiple independent consumers read the same stream. Built for millions of events per second.

Core concepts

const { Kafka } = require('kafkajs');
const kafka = new Kafka({ brokers: ['kafka:9092'] });

// Producer — OrderService publishes an event
const producer = kafka.producer();
await producer.connect();
await producer.send({
  topic:    'order-events',
  messages: [{ key: 'order-123', value: JSON.stringify({ orderId: 123, status: 'placed' }) }]
});

// Consumer — EmailService subscribes independently
const consumer = kafka.consumer({ groupId: 'email-service' });
await consumer.connect();
await consumer.subscribe({ topic: 'order-events', fromBeginning: false });
await consumer.run({
  eachMessage: async ({ message }) => {
    const event = JSON.parse(message.value.toString());
    await sendOrderConfirmationEmail(event);
  }
});
Kafka vs simple queue (SQS/RabbitMQ): In SQS, a message is deleted once consumed. In Kafka, messages are retained (default 7 days). A new analytics service can join and replay all historical events from the beginning. This is why Kafka powers event sourcing, audit logs, and real-time analytics pipelines.

4Redis — the Swiss Army knife

Redis is an in-memory data store. Microsecond reads. Used for caching, session storage, rate limiting, leaderboards, pub/sub, and job queues.

const redis = require('ioredis');
const client = new redis('redis://localhost:6379');

// Cache-aside pattern — check Redis first, DB second
async function getUser(id) {
  const key    = 'user:' + id;
  const cached = await client.get(key);
  if (cached) return JSON.parse(cached);   // cache hit: ~0.1ms

  const user = await db.findUser(id);       // cache miss: ~10ms
  await client.setex(key, 3600, JSON.stringify(user)); // TTL 1hr
  return user;
}

// Rate limiting — max 100 requests per IP per minute
async function rateLimit(ip) {
  const key   = 'rate:' + ip;
  const count = await client.incr(key);
  if (count === 1) await client.expire(key, 60);
  return count > 100; // true = blocked
}
Common use cases
Session storage
API rate limiting
Leaderboards (Sorted Sets)
Job queues (Bull/BullMQ)
Pub/sub for real-time features
Redis data structures
String: cache, counters
Hash: user session objects
List: activity feeds, queues
Set: unique visitors, tags
Sorted Set: leaderboards, scores
Important: Redis is in-memory. Data can be lost if Redis restarts (unless persistence is enabled). Never use Redis as your only store for critical data — it complements your primary database, not replaces it.

5Microservices vs Monolith

Monolith
One codebase, one deployment, one database. Simple to build, test, and debug. Hard to scale one hot component independently. Right for most products until ~50 engineers.
Microservices
Each service is independent: own codebase, deployment, database. Teams move independently. Operationally complex — distributed tracing, service discovery, network calls everywhere.
CTO advice: Don't start with microservices. Every successful microservices architecture started as a monolith. Split only when you have a proven, specific scale or team ownership problem. Premature microservices is one of the most common early-stage engineering mistakes.

6CAP theorem

In any distributed system, during a network partition, you must choose between Consistency and Availability. You cannot have both simultaneously. Partition tolerance is not optional in a real network.

Consistency (C)
Every read returns the most recent write or an error. All nodes see the same data at the same time.
Availability (A)
Every request gets a non-error response. The system stays operational even if the data might be slightly stale.
Partition tolerance (P)
System works even when network failures split it into isolated parts. Required for any distributed system.
DatabaseChoiceWhen partitioned…
PostgreSQL, MySQLCPReturns error rather than stale data
Cassandra, DynamoDBAPReturns possibly-stale data, stays up
MongoDBCP (default)Configurable per-operation
Redis ClusterAPMay lose recent writes

7System design patterns

Load balancing
Distribute requests across multiple instances. AWS ALB, Nginx, HAProxy. Round-robin or least-connections. Health checks remove failed nodes automatically.
Caching layers
CDN → Redis → DB query cache. Each layer reduces load on the next. A CDN cache hit costs microseconds; a DB query costs milliseconds.
Rate limiting
Protect your API from abuse. Redis sliding-window counters. 100 req/min per IP. Return 429 Too Many Requests. Essential before going public.
Circuit breaker
If downstream service B is failing, stop calling it immediately. Return a cached fallback. Prevents cascade failures from taking down your whole system.
Database sharding
Split one large table across multiple DB servers by a shard key (user_id % N). Each shard holds a subset of the data. Used at extreme scale.
Event sourcing
Store all state changes as immutable events rather than overwriting state. Enables full audit history, time travel debugging, and event replay.

8The CTO architecture lens

Questions to ask about any distributed architecture:

• Where is the actual bottleneck? (CPU? DB? Network? I/O?)
• What happens when any single component goes down?
• Can we replay events if a consumer fails mid-processing?
• How do we trace a single request across 5 microservices?
• Are we solving a problem we actually have, or one we imagine at scale?
• Could a well-indexed Postgres query and Redis cache solve this instead of Kafka?
• What's the operational burden of this choice on a 5-person team?
The rule of three: Add architectural complexity only when you've hit the limit three times. Hit the DB wall? Add caching. Hit caching limits? Add read replicas. Hit replica limits? Consider sharding. Skip levels and you're over-engineering.
