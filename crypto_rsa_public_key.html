🔑 RSA — Public-Key Encryption & Signatures

The first practical public-key system (1977) and still everywhere. A public key anyone can use to lock a message or check a signature; a private key only you hold to unlock or sign. Its security rests on one fact: multiplying two huge primes is easy, but factoring the result back apart is not.

Part of the Cryptography topic. Where ECDH lets two parties agree a secret, RSA lets anyone encrypt to a published public key, or lets a holder sign so anyone can verify. Same key-pair idea, different mechanics — and a useful contrast to elliptic curves.

1The padlock idea

Analogy. Imagine you hand out open padlocks to the whole world but keep the only key. Anyone can put a message in a box and snap your padlock shut — but only you, with the key, can open it. The open padlock is your public key; the key is your private key.

Two operations fall out of one key pair:

  • Encryption: anyone encrypts with your public key; only your private key decrypts.
  • Signing: you sign with your private key; anyone verifies with your public key. (The mirror image — and proof you wrote it.)

2The math, small enough to check by hand

RSA is built from modular exponentiation and one hard problem: factoring. Pick two primes p and q; their product n = p·q is public. Recovering p and q from a 2048-bit n is infeasible — and without them you can't derive the private key. Here's the whole scheme with tiny numbers:

JavaScript · runnable
// Tiny RSA you can verify by hand
const modpow = (b, e, m) => { let r = 1n; b %= m; while (e > 0n) { if (e & 1n) r = r * b % m; b = b * b % m; e >>= 1n; } return r; };
const egcd = (a, b) => { let or = a, r = b, os = 1n, s = 0n; while (r) { const q = or / r; [or, r] = [r, or - q * r]; [os, s] = [s, os - q * s]; } return [or, os]; };
const modinv = (e, phi) => { let [g, x] = egcd(e, phi); return ((x % phi) + phi) % phi; };

const p = 11n, q = 13n;        // two secret primes
const n = p * q;                // public modulus = 143
const phi = (p - 1n) * (q - 1n);  // Euler's totient = 120 (secret)
const e = 7n;                   // public exponent (coprime with phi)
const d = modinv(e, phi);        // private exponent = e⁻¹ mod phi
console.log(`public key  (e, n) = (${e}, ${n})`);
console.log(`private key (d, n) = (${d}, ${n})`);

const m = 9n;                   // the message: any number < n
const c = modpow(m, e, n);       // ENCRYPT with the public key
const back = modpow(c, d, n);    // DECRYPT with the private key
console.log(`encrypt: ${m}^${e} mod ${n} = ${c}`);
console.log(`decrypt: ${c}^${d} mod ${n} = ${back}`);
console.log(`round-trip works? ${m === back}`);

Encrypting is c = mᵉ mod n; decrypting is m = cᵈ mod n. They undo each other because of how d is chosen relative to e and φ(n). An attacker who could factor n into p·q could recompute φ and then d — which is exactly why n must be enormous.

3Real RSA — big keys and proper padding

Real RSA uses 2048–4096-bit keys, and never the raw "textbook" formula above. Plain RSA is deterministic and malleable — the same message always encrypts to the same ciphertext, which leaks information and enables attacks. Production encryption always adds OAEP padding (randomized). Here's the real thing via WebCrypto:

JavaScript · runnable (WebCrypto)
// Real RSA-2048 with OAEP padding — encrypt to a public key
const kp = await crypto.subtle.generateKey(
  { name: 'RSA-OAEP', modulusLength: 2048,
    publicExponent: new Uint8Array([1, 0, 1]), hash: 'SHA-256' },
  true, ['encrypt', 'decrypt']);

const msg = new TextEncoder().encode('a short secret');
const ct = await crypto.subtle.encrypt({ name: 'RSA-OAEP' }, kp.publicKey, msg);
const pt = await crypto.subtle.decrypt({ name: 'RSA-OAEP' }, kp.privateKey, ct);
console.log('ciphertext bytes:', new Uint8Array(ct).length, '(fixed 256 for RSA-2048)');
console.log('decrypted       :', new TextDecoder().decode(pt));
console.log('⚠ max plaintext ≈ 190 bytes — RSA wraps KEYS, not bulk data');
RSA can only encrypt tiny data — smaller than the key size. So it's never used for bulk data directly; instead you do hybrid encryption: encrypt the data with AES-GCM, then RSA-encrypt the AES key. That's the same shape as ECIES, just with RSA doing the key-wrap. For agreeing session keys, ephemeral ECDH is usually preferred (smaller, faster, forward secrecy).

4Signatures — the key pair used in reverse

Swap the roles and RSA proves authorship: sign a message's hash with the private key, and anyone can verify with the public key. Only the private-key holder could have produced it, and any change breaks it. Modern RSA signing uses PSS padding:

JavaScript · runnable (WebCrypto)
// RSA-PSS: sign with the PRIVATE key, verify with the PUBLIC key
const kp = await crypto.subtle.generateKey(
  { name: 'RSA-PSS', modulusLength: 2048,
    publicExponent: new Uint8Array([1, 0, 1]), hash: 'SHA-256' },
  true, ['sign', 'verify']);

const data = new TextEncoder().encode('ship release v2');
const sig = await crypto.subtle.sign({ name: 'RSA-PSS', saltLength: 32 }, kp.privateKey, data);
const check = (d) => crypto.subtle.verify({ name: 'RSA-PSS', saltLength: 32 }, kp.publicKey, sig, d);
console.log('signature bytes :', new Uint8Array(sig).length);
console.log('verify original :', await check(data));
console.log('verify tampered :', await check(new TextEncoder().encode('ship release v3')));

Signature verification is what makes TLS certificates, signed software updates, and package registries trustworthy. (Digital signatures get their own module next, including the elliptic-curve variants ECDSA/EdDSA.)

5RSA vs elliptic curve — why EC often wins now

Security levelRSA key sizeElliptic-curve key size
112-bit2048224
128-bit3072256 (P-256)
192-bit7680384
256-bit15360521

RSA is simple to understand and still fine for signatures and certificates, but its keys are ~12× larger than EC for the same strength, its operations are heavier, and static RSA key-transport lacks forward secrecy. That's why modern TLS favors ephemeral ECDH for key exchange and increasingly EC signatures — though RSA remains extremely common for interop.

6Pitfalls

  • Never textbook RSA. Always use OAEP (encryption) or PSS (signatures). Unpadded RSA has a long history of breaks.
  • Key size ≥ 2048 bits (3072+ for long-term). 1024-bit RSA is considered broken.
  • Don't encrypt bulk data with RSA — wrap an AES key instead (hybrid).
  • Signing ≠ encryption keys. Use separate key pairs for separate purposes.
