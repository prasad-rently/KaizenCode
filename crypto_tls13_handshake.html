🔗 TLS 1.3 Handshake — The Capstone

Every HTTPS connection you make runs this. TLS 1.3 is where the whole series comes together: elliptic-curve key agreement, a certificate signature, a hash-based key schedule, and authenticated encryption — combined into one fast, forward-secret handshake.

This module assembles the previous six. If a piece here feels unfamiliar, its own module explains it: ECDH, Hashing, HKDF, AES-GCM, signatures, and RSA.

1The 1-RTT handshake at a glance

TLS 1.3 sets up a secure channel in a single round trip. The client guesses the key-exchange method and sends its key share immediately; the server replies with its own share, its certificate, and a proof — and the very next message is already encrypted.

   CLIENT                                        SERVER
   ClientHello  ─────────────────────────────▶
   + key_share (ephemeral ECDHE public)
                ◀─────────────────────────────  ServerHello
                                                 + key_share (ephemeral ECDHE public)
                                                 + Certificate
                                                 + CertificateVerify  (signature)
                                                 + Finished
   {both sides derive the same keys via HKDF}
   Finished     ─────────────────────────────▶
   {encrypted application data}  ◀───────────▶  {encrypted application data}

3See it: a whole handshake in the browser

This is a simplified but faithful TLS-1.3-style handshake — ECDHE key agreement, a transcript hash, an HKDF-derived traffic key, a server certificate signature, and an encrypted application record — all running live:

JavaScript · runnable (WebCrypto)
// A simplified TLS 1.3 handshake, assembled from the whole series
const TE = new TextEncoder(), TD = new TextDecoder();
const hex = u => [...new Uint8Array(u)].map(x => x.toString(16).padStart(2, '0')).join('');
const gk = () => crypto.subtle.generateKey({ name: 'ECDH', namedCurve: 'P-256' }, true, ['deriveBits']);

// 1) ClientHello / ServerHello — each sends an EPHEMERAL key share
const client = await gk(), server = await gk();
const cShare = new Uint8Array(await crypto.subtle.exportKey('raw', client.publicKey));
const sShare = new Uint8Array(await crypto.subtle.exportKey('raw', server.publicKey));

// 2) Both derive the SAME ECDHE shared secret
const shared = await crypto.subtle.deriveBits({ name: 'ECDH', public: server.publicKey }, client.privateKey, 256);
console.log('handshake secret :', hex(shared).slice(0, 24) + '…');

// 3) Transcript hash binds the keys to THIS exact handshake
const transcript = await crypto.subtle.digest('SHA-256', new Uint8Array([...cShare, ...sShare]));

// 4) Server authenticates: signs the transcript with its certificate key
const cert = await crypto.subtle.generateKey({ name: 'ECDSA', namedCurve: 'P-256' }, true, ['sign', 'verify']);
const proof = await crypto.subtle.sign({ name: 'ECDSA', hash: 'SHA-256' }, cert.privateKey, transcript);
console.log('client verifies server cert:', await crypto.subtle.verify({ name: 'ECDSA', hash: 'SHA-256' }, cert.publicKey, proof, transcript));

// 5) HKDF key schedule → AES-GCM traffic key
const hk = await crypto.subtle.importKey('raw', shared, 'HKDF', false, ['deriveKey']);
const key = await crypto.subtle.deriveKey(
  { name: 'HKDF', hash: 'SHA-256', salt: new Uint8Array(transcript), info: TE.encode('tls13 app traffic') },
  hk, { name: 'AES-GCM', length: 256 }, false, ['encrypt', 'decrypt']);

// 6) Encrypted application record: client → server
const iv = crypto.getRandomValues(new Uint8Array(12));
const record = await crypto.subtle.encrypt({ name: 'AES-GCM', iv }, key, TE.encode('GET /account HTTP/2'));
console.log('encrypted record :', hex(record).slice(0, 24) + '…');
console.log('server decrypts  :', TD.decode(await crypto.subtle.decrypt({ name: 'AES-GCM', iv }, key, record)));

Try it yourself — open a secure channel

Type a request and run a fresh handshake. Each run generates brand-new ephemeral keys, verifies the server's certificate, derives a traffic key, and encrypts your message. Run it twice: the shared secret and record change every time — that's forward secrecy, and it's why capturing today's traffic can't decrypt tomorrow's.

4The key schedule (why HKDF, not just one key)

TLS 1.3 doesn't derive a single key — it runs an HKDF chain that produces a family of keys, each bound to a stage and the running transcript hash: early-secret → handshake-secret → master-secret, and from those, separate client/server handshake keys and application keys. Distinct keys per direction and per phase mean a compromise of one never exposes another — the domain-separation idea from the HKDF module, applied rigorously.

5Security by deletion — what 1.3 removed

TLS 1.3's biggest security win was taking things away. Gone: static RSA key transport (no forward secrecy), static Diffie–Hellman, CBC-mode ciphers, RC4, MD5/SHA-1, compression, and renegotiation. What's left is a small set of modern, AEAD-only, forward-secret options — far less surface for a downgrade or misconfiguration.

  • Always forward-secret — key exchange is ephemeral ECDHE only.
  • AEAD only — AES-GCM or ChaCha20-Poly1305; no unauthenticated modes.
  • Faster — 1-RTT by default (vs 2-RTT in TLS 1.2), and optional 0-RTT resumption.
0-RTT has a caveat. The optional "0-RTT" early-data feature lets a resumed connection send data in the very first message — but that early data is replayable by an attacker. It must only carry idempotent requests. A neat reminder that even the best protocol has trade-offs you must understand.

6Authentication — how it closes the ECDH MITM gap

The ECDH module ended on a warning: raw key agreement is confidential but not authenticated, so an active attacker can sit in the middle. TLS 1.3 closes that gap with the CertificateVerify step: the server signs the handshake transcript with the private key bound to its certificate, and the client checks that signature and the certificate chain up to a trusted CA. A man-in-the-middle can swap key shares, but can't forge that signature — so the handshake fails. Key agreement gives secrecy; the certificate signature gives identity.

7Where this runs

  • Every HTTPS request — the 🔒 in your address bar is a completed TLS handshake.
  • HTTP/3 (QUIC) — uses the TLS 1.3 handshake for its encryption.
  • Email, APIs, databases, VPNs — TLS underpins secure transport almost everywhere.
