🪢 The Safety Net

Tests aren't about catching bugs today — they're what lets you change code fearlessly tomorrow. The net that catches you when you refactor.

1The test pyramid

Write many cheap, fast tests and few slow, expensive ones. Shape your suite like a pyramid, not an ice-cream cone.

LayerTestsSpeedHow many
UnitOne function/module in isolation⚡ msLots
IntegrationModules together (e.g. route + DB)🐢 100s msSome
E2EWhole system via the real UI/API🐌 secondsA few
Mobile analogy: unit ≈ testing a single view-model, integration ≈ testing a screen with its store, E2E ≈ XCUITest / Detox driving the real app.

2Build your own test runner

Frameworks (Jest, Vitest) feel like magic until you see how little they are. Here's a working describe / it / expect in ~15 lines — run it.

JavaScript · runnable
// A minimal test framework
let pass = 0, fail = 0;
const it = (name, fn) => {
  try { fn(); console.log('✓ ' + name); pass++; }
  catch (e) { console.log('✗ ' + name + ' — ' + e.message); fail++; }
};
const expect = (got) => ({
  toBe: (want) => { if (got !== want)
    throw new Error(`expected ${want}, got ${got}`); }
});

// the function under test
const add = (a, b) => a + b;

it('adds positives', () => expect(add(2, 3)).toBe(5));
it('adds negatives', () => expect(add(-1, -1)).toBe(-2));
it('catches a bug', () => expect(add(2, 2)).toBe(5)); // fails on purpose
console.log(`\n${pass} passed, ${fail} failed`);

3Test-Driven Development (Red → Green → Refactor)

  1. Red — write a failing test for the behaviour you want.
  2. Green — write the simplest code to make it pass.
  3. Refactor — clean it up with the test guarding you.

Watch the cycle: the test exists first and fails, then we make it pass.

JavaScript · runnable
// RED: spec for a slugify() that doesn't exist yet
const assert = (cond, msg) => console.log((cond ? '✓ ' : '✗ ') + msg);

// GREEN: simplest implementation that satisfies the specs
const slugify = (s) => s.toLowerCase().trim()
  .replace(/[^a-z0-9]+/g, '-')
  .replace(/^-|-$/g, '');

assert(slugify('Hello World') === 'hello-world', 'spaces → dashes');
assert(slugify('  Trim Me!  ') === 'trim-me', 'trims + strips punctuation');
assert(slugify('A & B') === 'a-b', 'collapses separators');

4Mocking & test doubles

Don't hit the real database or network in a unit test. Replace dependencies with fakes you control, so tests are fast and deterministic.

JavaScript · runnable
// Inject the dependency, then fake it in the test
async function getUserName(id, db) {       // db is injected
  const user = await db.findUser(id);
  return user ? user.name : 'unknown';
}

// a mock db — no real database needed
const fakeDb = {
  calls: 0,
  findUser(id) { this.calls++; return { id, name: 'Gokul' }; }
};

console.log(await getUserName(42, fakeDb));   // 'Gokul'
console.log('db was called', fakeDb.calls, 'time(s)'); // assert interactions

5Coverage — a guide, not a goal

Coverage tells you which lines ran during tests. Useful to find untested areas — useless as a target. 100% coverage of meaningless assertions tests nothing.

Chasing a coverage % is a trap. Test behaviour and edge cases (empty input, nulls, boundaries, errors), not lines. One good test beats ten that assert toBeDefined().
Shell · copy
# the real-world stack for the projects in this roadmap
npm i -D vitest                 # fast unit/integration runner
npx vitest                      # watch mode, re-runs on save
npx vitest --coverage           # see what's untested
npm i -D supertest              # integration-test Express routes
npx playwright test             # end-to-end against a real browser
Try it: Add Vitest to the Phase 1 API. Write a unit test for one pure function, then a Supertest integration test that hits POST /tasks and asserts a 201 plus the created body. That's the pyramid's bottom two layers in practice.
