✍️ Digital Signatures — ECDSA & EdDSA

Encryption keeps a message secret. A signature does the opposite job: it makes a message public but provable — anyone can confirm who wrote it and that not one byte has changed. Sign with your private key; the whole world verifies with your public key.

Part of the Cryptography topic — and the mirror image of RSA encryption. Encryption uses the recipient's public key to hide data; signing uses the sender's private key to prove authorship. Same key-pair idea, roles reversed.

1What a signature proves

Analogy — a wax seal that reshapes itself to the letter. A signature is bound to the exact content: change a single character and the seal no longer matches. It gives you three guarantees at once:

  • Authenticity — it came from the holder of the private key.
  • Integrity — the message hasn't been altered since signing.
  • Non-repudiation — the signer can't later deny it (only they hold the key).

Crucially, anyone can verify with the public key — unlike HMAC, where only holders of the shared secret can check. That's what makes signatures work for public trust: certificates, app stores, software updates.

2How it works

Two steps, mirroring encryption:

   SIGN (private key)                 VERIFY (public key)
   message → hash → sign(hash) = 𝗌𝗂𝗀   message → hash ┐
                                          𝗌𝗂𝗀 ────────┴→ valid? ✓ / ✗

You never sign the whole message directly — you hash it first (fast, fixed-size), then apply the private-key math to that hash. Verification re-hashes the received message and checks the signature against it with the public key. If the message changed, the hashes differ and verification fails.

3See it: ECDSA over P-256

ECDSA is the elliptic-curve signature scheme — same curves as ECDH, used for signing. Generate a key pair, sign a message, verify it, then tamper and watch it fail:

JavaScript · runnable (WebCrypto)
// ECDSA (P-256): sign with the PRIVATE key, verify with the PUBLIC key
const TE = new TextEncoder();
const kp = await crypto.subtle.generateKey(
  { name: 'ECDSA', namedCurve: 'P-256' }, true, ['sign', 'verify']);

const msg = TE.encode('transfer 5 BTC to Alice');
const sig = await crypto.subtle.sign({ name: 'ECDSA', hash: 'SHA-256' }, kp.privateKey, msg);

const check = (m) => crypto.subtle.verify({ name: 'ECDSA', hash: 'SHA-256' }, kp.publicKey, sig, m);
console.log('signature bytes :', new Uint8Array(sig).length, '(P-256 → 64)');
console.log('verify original message :', await check(msg));
console.log('verify tampered message :', await check(TE.encode('transfer 5 BTC to Eve')));

A 64-byte signature vs RSA's 256 bytes for comparable security — the same size advantage elliptic curve gives everywhere.

4EdDSA (Ed25519) — the modern default

EdDSA, usually over the Ed25519 curve, is the newer scheme now preferred for new systems. It's faster, has simple fixed-size keys, and is deterministic — it doesn't need a fresh random number per signature, which sidesteps ECDSA's most dangerous footgun (below).

JavaScript · runnable (WebCrypto)
// EdDSA (Ed25519): modern, deterministic, fast — recent browsers only
const TE = new TextEncoder();
try {
  const kp = await crypto.subtle.generateKey('Ed25519', true, ['sign', 'verify']);
  const msg = TE.encode('release build 42');
  const sig = await crypto.subtle.sign('Ed25519', kp.privateKey, msg);
  console.log('Ed25519 signature bytes:', new Uint8Array(sig).length);
  console.log('verify original :', await crypto.subtle.verify('Ed25519', kp.publicKey, sig, msg));
  console.log('verify tampered :', await crypto.subtle.verify('Ed25519', kp.publicKey, sig, TE.encode('release build 43')));
} catch (e) {
  console.log('Ed25519 not available in this browser:', e.name);
  console.log('(Chrome 137+, Safari 17+, Firefox 129+. The ECDSA demo above works everywhere.)');
}

5Signatures vs HMAC vs encryption

Signature (ECDSA/EdDSA/RSA)HMACEncryption
Who can produce it?Only the private-key holderAny shared-key holderAnyone (public-key) / key-holder
Who can verify it?Anyone (public key)Only shared-key holdersN/A
Proves authorship?Yes — non-repudiableYes, but any holder could haveNo
Hides the message?NoNoYes

Rule of thumb: public trust → signature (you publish a verifiable key); a shared secret between two parties → HMAC (cheaper, symmetric).

6Where you'll meet signatures

  • TLS certificates — a CA signs a site's certificate; your browser verifies the chain.
  • JWTs — ES256 (ECDSA) and EdDSA tokens are signed, so a server can trust them without a shared secret.
  • Software & OS updates — signed so a device installs only genuine, unaltered builds.
  • Crypto wallets — every blockchain transaction is an ECDSA/EdDSA signature authorizing a transfer.
  • Git & packages — signed commits/tags and signed package releases (npm provenance, Sigstore).
ECDSA's footgun: never reuse (or leak) the per-signature random k. ECDSA needs a fresh random nonce for every signature. Reuse the same k for two messages — or use a predictable one — and anyone can solve for your private key from the two signatures. This exact bug broke the Sony PlayStation 3's code signing in 2010. Mitigations: use deterministic ECDSA (RFC 6979), or prefer EdDSA, which is deterministic by design and immune to this class of mistake.
