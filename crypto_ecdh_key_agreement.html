🤝 ECDH — Elliptic-Curve Key Agreement

How two parties who have never met agree on a shared secret over a wide-open line — no password exchanged in advance, no eavesdropper able to reproduce it. The intuition, the math you can run by hand, and real browser-native demos.

First in the Cryptography topic — a growing series that explains one method at a time from scratch. Generic and framework-free: everything here is standard cryptography you'll meet in TLS, Signal, SSH, WireGuard, and hardware-backed key stores.

1The problem

Two parties — call them A and B — want a shared encryption key. But they've never shared a secret in advance, and everything they send crosses a line an attacker can read. How do they agree on a key an eavesdropper can't learn, using only public messages?

The answer is Diffie–Hellman key agreement (Whitfield Diffie & Martin Hellman, 1976). Its modern, elliptic-curve form is ECDH. The key insight: each side keeps its own, different private key and only ever publishes a public key derived from it — yet both independently arrive at the identical shared secret.

Key agreement ≠ key transport. Nobody encrypts a key and sends it. The shared secret is never transmitted at all — both ends compute the same value locally from public inputs. That's the whole trick.

2Intuition — mixing paint

Forget math for a second. Two people want a shared secret colour and have never met.

  • Public base colour everyone knows (including attackers): YELLOW.
  • A secretly picks BLUE · B secretly picks RED. These are their private keys — different, and never revealed.
  • Each mixes their private colour into yellow and sends the result out in the open: A sends GREEN (yellow+blue), B sends ORANGE (yellow+red). These are their public keys.
  • Each adds their own private colour to the colour they received: A does orange+blue, B does green+red. Both get the same BROWN = yellow+red+blue.
        PUBLIC                A (private BLUE)          B (private RED)
   base = YELLOW
                          YELLOW+BLUE = GREEN  ───▶  (B receives GREEN)
   (B sends) ORANGE ◀───  YELLOW+RED  = ORANGE
                          ORANGE+BLUE = BROWN        GREEN+RED = BROWN
                                │                          │
                                └──── SAME SECRET ─────────┘
                                     BROWN = YELLOW+RED+BLUE

   Eavesdropper saw YELLOW, GREEN, ORANGE  →  cannot produce BROWN
   (never had BLUE or RED, and you can't un-mix paint)

The one-way step is "un-mixing paint" — easy to mix, practically impossible to reverse. Real crypto swaps paint for arithmetic with the same property.

3The math, with tiny numbers

The number version uses modular exponentiation: easy one way, practically impossible to reverse (the discrete-logarithm problem). Meet Alice and Bob (cryptography's traditional stand-ins for "the two sides"). Public values everyone knows: a prime p = 23 and a base g = 5. Each picks their own private number, computes g^private mod p as their public key, then raises the received public key to their own private number. Run it and watch every step:

JavaScript · runnable
// Diffie–Hellman by hand — every step of the arithmetic shown
const p = 23n, g = 5n;   // public: prime + base
const a = 6n;              // Alice's private number
const b = 15n;             // Bob's private number

// compute (base^exp) mod m — and PRINT the full arithmetic
const step = (base, exp, m) => {
  const full = base ** exp;   // the whole power…
  const res  = full % m;      // …then reduce mod m
  console.log(`   ${base}^${exp} = ${full}  →  mod ${m} = ${res}`);
  return res;
};

console.log(`PUBLIC (everyone sees): p = ${p}, g = ${g}`);
console.log(`PRIVATE (secret):       Alice a = ${a}, Bob b = ${b}`);

console.log('\nStep 1 — each makes a public key = g^private mod p:');
const alicePublic = step(g, a, p);   // 5^6  mod 23
const bobPublic   = step(g, b, p);   // 5^15 mod 23
console.log(`   → Alice's public key = ${alicePublic}, Bob's = ${bobPublic}`);

console.log('\nStep 2 — Alice and Bob swap public keys over the open network.');

console.log('\nStep 3 — each raises the OTHER key to their own private number:');
const aliceSecret = step(bobPublic, a, p);   // 19^6 mod 23
const bobSecret   = step(alicePublic, b, p);   // 8^15 mod 23

console.log(`\n✓ Same shared secret? ${aliceSecret === bobSecret}  →  both got ${aliceSecret}`);
console.log('   (both equal g^(a·b) mod p = 5^90 mod 23 = 2 — and it never crossed the wire)');

Alice and Bob both land on the same number because each is really g^(alicePrivate·bobPrivate) mod p, just computed in a different order. An eavesdropper sees p, g, and both public keys but to recover either private number must solve 5^x mod 23 = (a public key) — trivial at p=23, but with a 256-bit curve it is beyond every computer on Earth.

4Why elliptic-curve DH

Here's the reassuring part: ECDH is the same handshake you just ran in §3 — swap public keys, both sides land on the same shared secret. Only the one-way math underneath changes. §3 used "raise a number to a power." ECDH uses "copy a point on a curve, over and over." You can understand the whole thing with no equations at all.

Picture a pond of stepping stones

Everyone shares one starting stone called G, and one fixed rule for turning any stone into the "next" one. Follow that rule to make copies of the starting stone: do it 3 times and you land on the stone we call 3·G; do it 7 times and you're on 7·G. The stones scatter all over the pond in no order you could guess:

   1·G      2·G      3·G      4·G      5·G   …   ← "G copied that many times"
  (5,1)    (6,3)   (10,6)   (3,1)   (9,16)  …   ← scattered, no pattern to follow
  • Your private key is just a secret whole number — how many copies you make.
  • Your public key is the stone you land on after that many copies.

Making copies forward is easy — keep applying the rule. Going backward is hopeless: hand someone the stone 7·G and they can't tell it took 7 copies, because the stones jump around with no trail to follow. Easy one way, impossible the other — the same "you can't un-mix paint" lock from the very first analogy.

The handshake — and why it works

Say Alice's secret number is 3 and Bob's is 7:

  1. Alice makes 3 copies of G and shares the result, 3·G. Bob shares 7·G.
  2. Alice takes Bob's stone and copies it her 3 times → 3·(7·G).
  3. Bob takes Alice's stone and copies it his 7 times → 7·(3·G).

They land on the exact same stone — because 3 × 7 and 7 × 3 are the same number, 21. Both quietly end up on 21·G, yet neither ever said their secret number out loud. The eavesdropper watched 3·G and 7·G go past but can't undo them back to 3 and 7, so 21·G stays out of reach. That's the entire handshake.

What the curve — and the "stones" — actually look like

On the left is the curve over ordinary numbers: the smooth, symmetric shape it's named for. On the right is the same curve over the field of 17 — it stops being a line and becomes a scatter of just 18 dots. Those dots are our stepping stones, and G = (5, 1) is where every copy starts.

See it on a real (tiny) curve

Below is a genuine elliptic curve, shrunk to just 17 points so it fits on screen — and written with plain names. The first box is the rule for combining two stones; you can skim right past it. The second box is the part to read. Press Run and watch: the copies scatter, and Alice and Bob end up on the same stone.

JavaScript · runnable
// ══ Box 1: the rule for combining two stones (real curve math — skim it) ══
const wrap = n => ((n % 17) + 17) % 17;        // stay on the "17-clock": keep numbers in 0…16
const undoMultiply = x => {                    // the number that multiplies x back to 1 — just try all
  for (let k = 1; k < 17; k++) if (wrap(x * k) === 1) return k;
};
const OFF = null;                               // the special "off the board" stone
function combine(P, Q) {                        // add two stones together → a third stone
  if (P === OFF) return Q;
  if (Q === OFF) return P;
  const [px, py] = P, [qx, qy] = Q;
  if (px === qx && wrap(py + qy) === 0) return OFF;
  const line = (px === qx && py === qy)
    ? wrap((3 * px * px + 2) * undoMultiply(2 * py))   // same stone → the tangent line
    : wrap((qy - py) * undoMultiply(qx - px));           // two stones → the line through them
  const x = wrap(line * line - px - qx);
  const y = wrap(line * (px - x) - py);
  return [x, y];
}

// ══ Box 2: the part to actually read ══
const G = [5, 1];                            // the starting stone everyone shares
// copies(n, stone) = that stone added to itself n times  →  n·stone
const copies = (n, stone) => {
  let here = stone;
  for (let i = 1; i < n; i++) here = combine(here, stone);
  return here;
};
const show = P => P === OFF ? 'off-board' : `(${P[0]}, ${P[1]})`;

// Where 1…8 copies of G land — watch the stones scatter
for (let n = 1; n <= 8; n++)
  console.log(n + '·G = ' + show(copies(n, G)) + (n === 3 ? '   ← Alice' : n === 7 ? '   ← Bob' : ''));

// The handshake — Alice's secret number is 3, Bob's is 7 (never shared)
const alicePublic = copies(3, G);         // = 3·G, sent in the open
const bobPublic   = copies(7, G);         // = 7·G, sent in the open
console.log('Alice shares 3·G =', show(alicePublic));
console.log('Bob   shares 7·G =', show(bobPublic));
console.log('Alice makes 3 copies of Bob’s stone  →', show(copies(3, bobPublic)));
console.log('Bob   makes 7 copies of Alice’s stone →', show(copies(7, alicePublic)));
console.log('Same stone! both are 21·G, because 3 × 7 = 7 × 3 ✓');

Real curves (P-256, Curve25519) are this exact idea, just with a pond of about 1077 stones instead of 17 — the same copying, an unimaginably bigger pond. Because there's no shortcut back through all those stones, elliptic curves need far smaller keys than RSA or classic DH for the same strength:

Security levelRSA / classic DHElliptic curve
112-bit2048-bit224-bit
128-bit3072-bit256-bit (P-256)
192-bit7680-bit384-bit
256-bit15360-bit521-bit

Smaller keys mean faster math, less CPU/battery, and smaller payloads on the wire — which is why P-256 / Curve25519 dominate modern protocols. (Source: NIST SP 800-57 comparable-strength figures.)

5From shared secret to an encrypted message (ECIES)

You don't encrypt with the raw shared secret. The standard flow — ECIES (Elliptic Curve Integrated Encryption Scheme) — is hybrid: use ECDH to agree a key, then a fast symmetric cipher to encrypt the data.

  • HKDF the shared secret into a clean AES key ("polish this into a proper key").
  • Encrypt the payload with AES-256-GCM — fast, and its auth tag detects tampering.

This is exactly how HTTPS/TLS works: asymmetric to agree a key, symmetric to move the bytes. Here's a real, end-to-end ECDH + HKDF + AES-GCM round-trip using the browser's own crypto.subtle — no libraries:

JavaScript · runnable (WebCrypto)
// Real P-256 ECDH in the browser, then ECIES-style encrypt/decrypt
const gen = () => crypto.subtle.generateKey(
  { name: 'ECDH', namedCurve: 'P-256' }, false, ['deriveBits']);
const alice = await gen();      // each party makes its OWN keypair
const bob   = await gen();
const hex = u => [...new Uint8Array(u)]
  .map(x => x.toString(16).padStart(2, '0')).join('');

// 1) Both derive the SAME 256-bit secret (own private + peer's public)
const bits = (priv, pub) =>
  crypto.subtle.deriveBits({ name: 'ECDH', public: pub }, priv, 256);
const sA = await bits(alice.privateKey, bob.publicKey);
const sB = await bits(bob.privateKey, alice.publicKey);
console.log('A secret:', hex(sA).slice(0, 24) + '…');
console.log('B secret:', hex(sB).slice(0, 24) + '…');
console.log('Match?', hex(sA) === hex(sB));

// 2) ECIES: HKDF the secret into an AES-256-GCM key, then encrypt
const hk = await crypto.subtle.importKey('raw', sA, 'HKDF', false, ['deriveKey']);
const aesKey = await crypto.subtle.deriveKey(
  { name: 'HKDF', hash: 'SHA-256', salt: new Uint8Array(), info: new Uint8Array() },
  hk, { name: 'AES-GCM', length: 256 }, false, ['encrypt', 'decrypt']);
const iv = crypto.getRandomValues(new Uint8Array(12));
const msg = new TextEncoder().encode('API_KEY=super-secret-value');
const ct = await crypto.subtle.encrypt({ name: 'AES-GCM', iv }, aesKey, msg);
const pt = await crypto.subtle.decrypt({ name: 'AES-GCM', iv }, aesKey, ct);
console.log('ciphertext:', new Uint8Array(ct).length, 'bytes (unreadable)');
console.log('decrypted :', new TextDecoder().decode(pt));

6Ephemeral keys & forward secrecy

If at least one side generates a brand-new keypair for every exchange (an ephemeral key — the "E" in TLS's ECDHE), then every session has its own unique shared secret. Crack or leak one session's key and no other session is affected; an attacker who recorded past traffic still can't read it. This property is forward secrecy, and it's a big reason ephemeral ECDH replaced static RSA key-transport in TLS 1.3.

7ECDH vs RSA — when to reach for which

ECDH (key agreement)RSA (key transport)
How a key is sharedBoth sides compute it; never sentOne side encrypts a key and sends it
Key size @128-bit256-bit3072-bit
Forward secrecyFree (with ephemeral keys)No — leak the static key, all past traffic falls
Direct payload encryptionN/A — always hybrid (ECIES)Tiny only (< key size); needs hybrid anyway
Padding pitfallsNone of RSA's classPKCS#1 v1.5 oracles; needs OAEP + care
Hardware key storesSupported (EC P-256)Often not — e.g. iOS Secure Enclave is EC-only

Bottom line: for agreeing a session key, ECDH is smaller, faster, gives forward secrecy for free, and is the only asymmetric primitive some secure hardware supports. RSA still appears for signatures and legacy interop, but for key exchange, ephemeral ECDH is the modern default.

8Where you'll meet ECDH in the wild

  • TLS 1.3 / HTTPS — ECDHE is the default handshake; every request you make already uses it.
  • Signal / WhatsApp — X3DH and the Double Ratchet are built on chained ECDH (Curve25519).
  • SSH & WireGuard — ECDH key exchange (Curve25519) underpins the tunnels.
  • Hardware-backed key delivery — a device keeps a non-extractable EC private key in a Secure Enclave (iOS) / StrongBox (Android); a server sends an ephemeral public key so only that one physical device can derive the secret and decrypt the payload.
Crucial caveat: ECDH gives confidentiality, not authentication. On its own it agrees a secret with whoever is on the other end — it does not prove who that is. If the public keys aren't authenticated, an active attacker can sit in the middle, run one ECDH with each side, and relay (a man-in-the-middle attack). Real protocols authenticate the public keys — with certificates (TLS), pre-shared identity keys (Signal), or device attestation — on top of ECDH. Never ship raw ECDH and assume you know who you're talking to.
