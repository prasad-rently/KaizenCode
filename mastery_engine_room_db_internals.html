⚙️ The Engine Room

Phase 2 taught you to use a database. The Engine Room shows you how one actually works underneath — indexes, transactions, isolation, the write path — so you can diagnose the slow query and the mysterious data race instead of guessing.

1Why an index is fast (B-trees)

Without an index, finding a row means scanning every row — O(n). An index keeps keys sorted in a balanced B-tree, so lookups are O(log n): a handful of hops even in a billion rows. Feel the gap:

JavaScript · runnable
const N = 1_000_000;
const sorted = Array.from({ length: N }, (_, i) => i); // an "index"
const target = 987654;

// ❌ full table scan — check every row
let steps = 0;
for (let i = 0; i < N; i++) { steps++; if (sorted[i] === target) break; }
console.log('seq scan steps: ', steps);

// ✅ B-tree-style binary search on the sorted index
let lo = 0, hi = N - 1, bsteps = 0;
while (lo <= hi) { bsteps++; const mid = (lo + hi) >> 1;
  if (sorted[mid] === target) break;
  sorted[mid] < target ? lo = mid + 1 : hi = mid - 1; }
console.log('index lookup steps:', bsteps, '— that is O(log n)');
Indexes aren't free. Every index must be updated on every INSERT/UPDATE/DELETE, and it costs storage. Index the columns you filter/join/sort on — not every column.

2Reading the query plan

EXPLAIN ANALYZE is how you see what the database actually did. The word you're hunting for is Seq Scan on a big table — it usually means a missing index.

SQL · copy
EXPLAIN ANALYZE SELECT * FROM orders WHERE user_id = 42;

-- ❌ before an index:
--   Seq Scan on orders  (cost=0.00..18334) rows=1  (actual time=142ms)

CREATE INDEX idx_orders_user ON orders(user_id);

-- ✅ after:
--   Index Scan using idx_orders_user  (actual time=0.03ms)  ← 4000x faster

3The write path — durability via the WAL

How does a DB survive a crash mid-write? It writes changes to an append-only Write-Ahead Log first (fast sequential disk write) and only later updates the actual data pages. On restart, it replays the WAL to recover. The "D" in ACID.

You've seen this before: the WAL is exactly event sourcing (The Streaming Mind) applied to storage — the durable log is the source of truth, and the table is a materialized view of it. Kafka's log and Redis' AOF are the same idea.

4Transactions & ACID

A transaction groups statements into an all-or-nothing unit. Atomic (all or none), Consistent (rules hold), Isolated (concurrent txns don't corrupt each other), Durable (survives crashes).

SQL · copy
BEGIN;
  UPDATE accounts SET balance = balance - 100 WHERE id = 1;
  UPDATE accounts SET balance = balance + 100 WHERE id = 2;
COMMIT;  -- both happen, or (on error) ROLLBACK undoes everything
-- money is never created or destroyed mid-transfer

5Isolation levels & their anomalies

Full isolation is expensive, so databases offer levels that trade correctness for speed. Each weaker level permits specific anomalies:

LevelAllows
Read UncommittedDirty reads (see uncommitted data)
Read Committed (PG default)Non-repeatable reads
Repeatable ReadPhantom reads
SerializableNothing — as if txns ran one at a time

Simulate a non-repeatable read: the same query returns different answers inside one transaction because another committed in between.

JavaScript · runnable
let row = { price: 100 };            // the committed database row

// Transaction A reads twice; Transaction B commits in between
const readA1 = row.price;               // A: first read
row = { price: 130 };                 // B: UPDATE ... COMMIT
const readA2 = row.price;               // A: same query, second read

console.log('read #1:', readA1);
console.log('read #2:', readA2, readA1 !== readA2 ? '← non-repeatable read!' : '');
console.log('fix → REPEATABLE READ / SERIALIZABLE isolation, or SELECT ... FOR UPDATE');
MVCC: Postgres avoids locking readers by keeping multiple versions of a row (Multi-Version Concurrency Control). Readers see a consistent snapshot; writers create new versions. It's why "readers don't block writers" — and why VACUUM exists to clean old versions.

6Connection pooling

Opening a DB connection is expensive (TCP + TLS + auth). Each Postgres connection also costs real server memory, so thousands of them crush it. A pool keeps a small set of warm connections and hands them out — the single highest-leverage DB perf fix in most apps.

JavaScript · copy
import { Pool } from 'pg';
const pool = new Pool({ max: 10 });  // reuse up to 10 warm connections

// borrow → query → auto-return to the pool
const { rows } = await pool.query('SELECT * FROM users WHERE id = $1', [42]);
// use a bounded pool (see The Juggler) so a spike can't open 10k connections
Try it: On the Phase 2 database, run EXPLAIN ANALYZE on a WHERE query over a large table, note the Seq Scan and time, add the index, and re-run to see the Index Scan. Then wrap a two-step balance transfer in BEGIN/COMMIT and prove a forced error rolls both steps back.
